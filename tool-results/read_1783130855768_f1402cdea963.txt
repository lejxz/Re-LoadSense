    26Ôćĺ## 1. What the original got right
    27Ôćĺ
    28ÔćĺBefore cataloguing failures, it is only fair to acknowledge the original's genuine strengths.
    29ÔćĺA 48-hour hackathon that shipped a working end-to-end demo ÔÇö synthetic fleet simulator, live
    30Ôćĺmap, operator console, alert workflow, and a chatbot ÔÇö is a real achievement. Specifically:
    31Ôćĺ
    32Ôćĺ- **Problem framing was excellent.** The "how full is the next jeepney?" gap is real,
    33Ôćĺ  underserved, and aligns cleanly with UN SDG 9 and SDG 11. The dual-layer (edge + cloud)
    34Ôćĺ  mental model is the correct architecture for this problem class.
    35Ôćĺ- **The four-tier occupancy taxonomy** (­ččó available / ­ččí filling / ­čö┤ at capacity / ­čö┤-blink
    36Ôćĺ  overloaded) is intuitive, actionable, and maps cleanly to both a windshield LED and a map
    37Ôćĺ  marker. This design choice survives into Re-LoadSense unchanged.
    38Ôćĺ- **Operator-first alert verification** (ack Ôćĺ verify Ôćĺ false-alarm / escalate) is a
    39Ôćĺ  thoughtful safety pattern that avoids black-box auto-escalation. It reflects genuine
    40Ôćĺ  domain understanding of transit operations.
    41Ôćĺ- **ASEAN expansion intent** ÔÇö seeding route data for PH / ID / MY / TH / VN and planning a
    42Ôćĺ  GTFS import path ÔÇö showed the right ambition, even though the execution was uneven.
    43Ôćĺ- **Contract-faithful simulation as a demo strategy** is, in principle, the right call for a
    44Ôćĺ  hackathon: produce the same data shape a real deployment would emit, so downstream code is
    45Ôćĺ  real even if the upstream is faked. (The execution of this principle was flawed ÔÇö see
    46Ôćĺ  ┬ž4 ÔÇö but the principle is sound and Re-LoadSense retains it, honestly labeled.)
    47Ôćĺ- **Config-driven thresholds** ÔÇö occupancy tiers, geofence radius, and capacity defaults
    48Ôćĺ  live in `config/project_config.json` rather than being hardcoded. This is good practice
    49Ôćĺ  and Re-LoadSense generalizes it into a full typed configuration system.
    50Ôćĺ- **Parameterized SQL** ÔÇö `sqlite_store.py` uses parameterized queries throughout. No SQL
    51Ôćĺ  injection vectors. This is a non-trivial discipline that many hackathon projects skip.
    52Ôćĺ
    53ÔćĺThese strengths are preserved in Re-LoadSense. Everything below is what must change.

    56Ôćĺ
    57Ôćĺ## 2. Backend failures
    58Ôćĺ
    59ÔćĺThe backend (`backend/app/`) is a FastAPI app with ~740 lines of route handlers, a 1,208-line
    60ÔćĺSQLite store, a 1,860-line god-module (`transit.py`), and five ÔÇö *five* ÔÇö chatbot files, two
    61Ôćĺof which are dead code.
    62Ôćĺ
    63Ôćĺ### 2.1 Five chatbot files, three of them dead
    64Ôćĺ
    65ÔćĺThe repo contains `chatbot.py` (Gemini), `no_API_chatbot.py` (heuristic), `ollama_chatbot.py`
    66Ôćĺ(Ollama), plus `phase2.py` and `compat.py` tangled into the same concern. Grep confirms that
    67Ôćĺ**`chatbot.py` and `ollama_chatbot.py` are never imported by the running application**. The
    68Ôćĺlive code path (`state.py:FleetStore.recommendation()`) explicitly bypasses the LLM paths and
    69Ôćĺcalls only `get_no_api_recommendation()`.
    70Ôćĺ
    71ÔćĺYet `requirements.txt` still ships `google-genai>=0.3.0` as a hard dependency, and the README
    72Ôćĺadvertises the Gemini chatbot as a feature. The Gemini file also contains a live bug
    73Ôćĺ(`chatbot.py:69`) papered over with a comment that literally says *"wait, should be
    74Ôćĺdestination string"* ÔÇö a developer's mid-fix note shipped to production.
    75Ôćĺ
    76Ôćĺ**Remediation:** One chatbot module with a strategy interface. Provider selection is a config
    77Ôćĺor feature-flag decision, not five copy-pasted files. Dead code is deleted, not archived.
    78Ôćĺ
    79Ôćĺ### 2.2 Zero authentication or authorization
    80Ôćĺ
    81ÔćĺNo endpoint has any auth. Anyone who can reach the API can:
    82Ôćĺ
    83Ôćĺ- `POST /api/database/reset` ÔÇö wipe everything (then auto-seeded with fake drivers).
    84Ôćĺ- `POST /api/alerts/reset`, `POST /api/demand/reset` ÔÇö wipe alerts / demand artifacts.
    85Ôćĺ- `DELETE /api/routes/{route}`, `DELETE /api/vehicles/{vehicle_id}` ÔÇö silent deletion.
    86Ôćĺ- `POST /api/vehicles`, `POST /api/routes`, `POST /api/alerts` ÔÇö inject arbitrary data.
    87Ôćĺ- Connect to `WS /ws/telemetry` and inject fake vehicle telemetry.
    88Ôćĺ- `POST /api/chatbot` ÔÇö potentially expensive NLP/LLM calls.
    89Ôćĺ
    90ÔćĺThere are no API keys, no JWT, no OAuth, no role separation between commuters, operators, and
    91Ôćĺadmins. The static frontend is served from the same origin, so CSRF is also a concern for
    92Ôćĺevery state-changing endpoint.
    93Ôćĺ
    94Ôćĺ**Remediation:** Re-LoadSense ships auth in the first vertical slice. JWT access tokens +
    95Ôćĺrefresh tokens, role-based access control (commuter / operator / admin / edge-device), per-
    96Ôćĺdevice credentials for edge telemetry, and CSRF protection on cookie-authenticated mutations.
    97Ôćĺ
    98Ôćĺ### 2.3 SQLite as the sole data store, with per-country fan-out
    99Ôćĺ
   100ÔćĺThe persistence layer is raw `sqlite3` (stdlib), no ORM, no async driver. Worse, it maintains
   101Ôćĺ**five separate SQLite files** ÔÇö one per country (PH, TH, VN, MY, ID) ÔÇö and almost every read
   102Ôćĺoperation iterates all five:
   103Ôćĺ
   104Ôćĺ- `list_vehicles()`, `list_incidents()`, `load_alerts()`, `load_vehicle_states()`,
   105Ôćĺ  `has_open_alert()`, `has_recent_vehicle_alert()`, `country_for_route()`, `route_exists()`,
   106Ôćĺ  `delete_vehicle()`, `load_route_polyline()`, `database_status()` all do
   107Ôćĺ  `for code in COUNTRY_CODES: ... query ...` then merge in Python.
   108Ôćĺ- `country_for_route()` is called on **every telemetry write** (`save_vehicle_state`), every
   109Ôćĺ  alert save, every chat save ÔÇö that's 5 `SELECT`s per write.
   110Ôćĺ- `has_open_alert()` and `has_recent_vehicle_alert()` iterate all 5 DBs per alert check, and
   111Ôćĺ  these are called on **every** telemetry upsert (`state.py:196, 231`).
   112Ôćĺ- `database_status()` runs ~10 aggregate queries ├Ś 5 countries = **50 queries per call**.
   113Ôćĺ
   114ÔćĺAdditionally, module-level shared state is accessed across threads without locking:
   115Ôćĺ`_connections` dict, `_LAST_TELEMETRY_LOG_WRITE` dict (read-modify-write race), and
   116Ôćĺ`_ROUTE_CACHE` are all touched by both the Uvicorn request threadpool and the demo simulator
   117Ôćĺdaemon thread.
   118Ôćĺ
   119Ôćĺ**Remediation:** PostgreSQL as the primary OLTP store with a proper connection pool
   120Ôćĺ(asyncpg/SQLAlchemy async). Multi-tenancy via a `country_code` column + indexes + row-level
   121Ôćĺsecurity, not file-per-country. Redis for ephemeral live-fleet state and caching. SQLite
   122Ôćĺremains only as an edge-device local buffer.
   123Ôćĺ
   124Ôćĺ### 2.4 No service layer ÔÇö business logic in route handlers
   125Ôćĺ
   126Ôćĺ`api/routes.py` contains ~740 lines with inline business logic: `parse_route_file`,
   127Ôćĺ`parse_geojson_routes`, `parse_csv_routes`, `parse_gtfs_routes`, `validate_polyline`,
   128Ôćĺ`validate_imported_routes`, `simplify_polyline` (Ramer-Douglas-Peucker). The `get_fleet`
   129Ôćĺhandler merges DB vehicles with live state directly in the handler function. There is no
   130Ôćĺservice / repository separation.
   131Ôćĺ
   132Ôćĺ`core/transit.py` is a **1,860-line god-module** mixing: place search, geocoding, route
   133Ôćĺmatching, multi-leg routing, vehicle ranking, fare estimation, NLP pattern matching,
   134Ôćĺtranslation, fuzzy matching, and language detection. Massive single-responsibility violation.
   135Ôćĺ
   136Ôćĺ**Remediation:** Layered architecture: routes Ôćĺ services Ôćĺ repositories Ôćĺ models. `transit.py`
   137Ôćĺsplits into `geocoding`, `route_matching`, `multi_leg`, `fare`, `nlp`, `i18n` modules. Each
   138Ôćĺservice has an interface and is unit-testable in isolation.
   139Ôćĺ
   140Ôćĺ### 2.5 No input validation beyond type checks
   141Ôćĺ
   142ÔćĺPydantic models validate types but almost no constraints:
   143Ôćĺ
   144Ôćĺ- `Telemetry.latitude` / `longitude` are bare `float` ÔÇö a client can send `lat=999`.
   145Ôćĺ- `occupancy: int` has no `ge=0`.
   146Ôćĺ- `timestamp: str` accepts any string (should be `datetime`).
   147Ôćĺ- `vehicle_id`, `route`, `driver` ÔÇö `str` with no length limit.
   148Ôćĺ- `ChatQuery.history: Optional[list[dict]]` ÔÇö no schema for the dicts.
   149Ôćĺ- `SuggestionQuery.limit: int = 5` ÔÇö no upper bound (could be 10,000,000).
   150Ôćĺ- `POST /routes/import` reads the entire upload into memory with no size limit ÔÇö DoS vector.
   151Ôćĺ
   152Ôćĺ**Remediation:** Every Pydantic model uses `Field(..., ge=, le=, min_length=, max_length=,
   153Ôćĺpattern=)` constraints. File uploads are streaming-parsed with a hard size cap. A shared
   154Ôćĺ`validators` module centralizes reusable constraints (lat/lon bounds, ISO timestamps, vehicle
   155ÔćĺID patterns).
   156Ôćĺ
   157Ôćĺ### 2.6 No API versioning, no response models, no OpenAPI discipline
   158Ôćĺ
   159ÔćĺAll endpoints live under `/api` with no version prefix. No `response_model` on any endpoint,
   160Ôćĺso the auto-generated OpenAPI schema has no response definitions. No `tags`, so `/docs` is a
   161Ôćĺflat list. No `responses=` to document error codes. The README documents endpoints that
   162Ôćĺ**do not exist**: `/api/demand/forecast`, `/api/eta/{vehicle_id}`,
   163Ôćĺ`/api/alerts/{id}/false-alarm`, `/api/alerts/{id}/escalate`, `/ws/fleet`.
   164Ôćĺ
   165Ôćĺ**Remediation:** `/api/v1/` prefix. Every endpoint declares `response_model`, `tags`,
   166Ôćĺ`summary`, `description`, and `responses=`. OpenAPI spec is exported, committed, and diffed in
   167ÔćĺCI ÔÇö breaking schema changes fail the build. README is generated from the spec, not hand-
   168Ôćĺwritten.
   169Ôćĺ
   170Ôćĺ### 2.7 Hardcoded values and magic numbers (partial list)
   171Ôćĺ
   172Ôćĺ| Location | Value | Problem |
   173Ôćĺ|---|---|---|

   175Ôćĺ| `ollama_chatbot.py:10-11` | `OLLAMA_API_URL = "http://localhost:11434"`, `MODEL_NAME = "llama3.2:3b"` | Won't work in container |
   176Ôćĺ| `routes.py:107` | `route="04L"`, `time_of_day=8.0`, `traffic_factor=1.0` | PH-specific defaults leak into other countries |
   177Ôćĺ| `state.py:246` | `self._alerts = self._alerts[-100:]` | Magic 100 cap |
   178Ôćĺ| `state.py:249-255` | traffic-factor weights `0.9, 1.05, 1.2, 1.35` | Magic |
   179Ôćĺ| `sqlite_store.py:22` | `TELEMETRY_LOG_INTERVAL_SECONDS = 15` | Magic |

   187Ôćĺ**Remediation:** All thresholds, fares, model names, timeouts, and radii move into a typed
   188Ôćĺconfig system (pydantic-settings) with per-country overrides. Seed data is culturally neutral
   189Ôćĺor per-country appropriate.
   190Ôćĺ
   191Ôćĺ### 2.8 DEMO_MODE defaults to true
   192Ôćĺ
   193Ôćĺ`is_demo_mode()` (`config.py:58-61`) reads the `DEMO_MODE` env var and **defaults to "true"**.
   194ÔćĺThis means a production deployment, unless someone explicitly sets `DEMO_MODE=false`, will run
   195Ôćĺthe synthetic fleet simulator ÔÇö a daemon thread that pollutes `telemetry_logs` with fake
   196Ôćĺvehicle data every 3 seconds. The demo simulator lives inside `core/` (the production
   197Ôćĺpackage), not in `dev/` or `tools/`.
   198Ôćĺ
   199Ôćĺ**Remediation:** `DEMO_MODE` defaults to `false`. The simulator moves to `dev/simulator/` and
   200Ôćĺis launched explicitly. Production images do not contain simulator code.
   201Ôćĺ
   202Ôćĺ### 2.9 No structured logging, no metrics, no tracing
   203Ôćĺ
   204ÔćĺThe entire backend uses `print()` for logging (`chatbot.py:137`, `ollama_chatbot.py:211`,
   205Ôćĺ`sqlite_store.py:580`). No `logging` module, no structlog/loguru, no log levels, no JSON log
   206Ôćĺformat. No Prometheus metrics endpoint. No OpenTelemetry tracing. No Sentry. The `/health`
   207Ôćĺendpoint returns `{"status": "ok"}` with no DB check, no model-load check, no dependency
   208Ôćĺcheck ÔÇö no liveness/readiness split.
   209Ôćĺ
   210Ôćĺ**Remediation:** structlog with JSON output and correlation IDs. Prometheus `/metrics`
   211Ôćĺendpoint with RED metrics (rate, errors, duration) per route. OpenTelemetry traces with
   212Ôćĺauto-instrumentation for FastAPI, asyncpg, and HTTP clients. Sentry for error tracking.
   213Ôćĺ`/health` (liveness) and `/ready` (readiness) split, with readiness checking DB + cache +
   214Ôćĺmodel availability.
   215Ôćĺ
   216Ôćĺ### 2.10 No rate limiting
   217Ôćĺ
   218ÔćĺNo `slowapi` or equivalent. `/chatbot` could trigger expensive NLP regex matching (or LLM
   219Ôćĺcalls if wired). `/places` makes external HTTP calls to Photon with a 1.8 s timeout ÔÇö easy to
   220Ôćĺabuse. `/routes/import` parses uploaded files with no size limit.
   221Ôćĺ
   222Ôćĺ**Remediation:** Token-bucket rate limiting per IP and per API key, configurable per route.
   223ÔćĺThe chatbot and places endpoints get stricter limits. A middleware-based limiter backed by
   224ÔćĺRedis so it works across multiple workers.

   228Ôćĺ## 3. Frontend failures
   229Ôćĺ
   230ÔćĺThe frontend (`app/`) is vanilla HTML/CSS/JS ÔÇö three static HTML pages (`index.html`,
   231Ôćĺ`mobile.html`, `operator.html`), seven CSS files, eight JS files totaling ~3,605 lines, plus
   232Ôćĺ~181 KB of vendored Leaflet committed to the repo. No build step, no module system, no type
   233Ôćĺsafety, no framework, no tests.
   234Ôćĺ
   235Ôćĺ### 3.1 No build step and no module system
   236Ôćĺ
   237ÔćĺEight JS files served as raw text via classic `<script src>` tags. All top-level function
   238Ôćĺdeclarations become implicit `window` globals. Load order is critical and undocumented. A
   239Ôćĺfunction defined in `routes-admin.js` silently overrides the same-named function in `map.js`
   240Ôćĺ(both define `setupRecenterButtons`, `fitRoute`, `fitFleet`, `previewRoute`, `zoomVehicle`).
   241ÔćĺThe `operator.js` version of `renderRouteDirectory` references an **undefined variable
   242Ôćĺ`groupBy`** (`operator.js:282, 285`) ÔÇö it would throw `ReferenceError` if ever called, but
   243Ôćĺit's dead-but-overriding code from copy-paste.
   244Ôćĺ
   245Ôćĺ**Remediation:** TypeScript + Vite. ES modules with explicit imports/exports. Build step
   246Ôćĺgives minification, tree-shaking, code-splitting, and source maps. Load-order bugs become
   247Ôćĺimpossible.
   248Ôćĺ
   249Ôćĺ### 3.2 XSS risks via innerHTML and inline onclick
   250Ôćĺ
   251Ôćĺ~50+ `innerHTML` assignments. Most use `escapeHtml()`, but `escapeHtml` does **not** escape
   252Ôćĺbackticks or `$`, so interpolating its output into JS template literals is unsafe. Map popups
   253Ôćĺ(`map.js:197, 209`) use inline `onclick="showVehicleDetailsModal('${escapeHtml(...)}')"` ÔÇö
   254Ôćĺinside an HTML attribute, entities are decoded, so an apostrophe in `vehicle_id` breaks the JS
   255Ôćĺstring syntax. `mobile.js:127, 177` interpolate `route` and `vehicle_id` into `innerHTML`
   256Ôćĺ**without** `escapeHtml` at all ÔÇö currently benign only because the simulator uses safe
   257Ôćĺstrings.
   258Ôćĺ
   259Ôćĺ**Remediation:** React (or Svelte) with JSX ÔÇö XSS-safe by default because interpolation is
   260Ôćĺauto-escaped. No `dangerouslySetInnerHTML` without DOMPurify sanitization. No inline event
   261Ôćĺhandlers.
   262Ôćĺ
   263Ôćĺ### 3.3 No accessibility
   264Ôćĺ
   265Ôćĺ- No skip-to-content link.
   266Ôćĺ- No ARIA live regions for chat (screen readers don't announce new bot messages).
   267Ôćĺ- Custom searchable dropdown (`core.js:451`) has no `role="listbox"`/`role="option"`, no
   268Ôćĺ  `aria-expanded`/`aria-haspopup`, no keyboard arrow navigation.
   269Ôćĺ- Mobile bottom nav and operator tabs lack `role="tablist"`/`role="tab"`/`role="tabpanel"`.
   270Ôćĺ- Modals are `<div class="modal hidden">`, not `<dialog>`, with no `role="dialog"`, no
   271Ôćĺ  `aria-modal="true"`, no focus trap, no ESC handler.
   272Ôćĺ- Chat `<input>` has no `<label>`.
   273Ôćĺ- SVG icons mostly lack `aria-hidden="true"`.
   274Ôćĺ- `blink` animation has no `prefers-reduced-motion` override.
   275Ôćĺ- `.hintrc` **explicitly disables** `button-type`, `label`, and `select-name` a11y hints.
   276Ôćĺ
   277Ôćĺ**Remediation:** axe-core in the test suite (fails CI on violations). Headless UI primitives
   278Ôćĺ(Radix UI) for dropdowns/dialogs/tabs that ship with correct ARIA out of the box. Keyboard
   279Ôćĺnavigation tested with Playwright. `prefers-reduced-motion` respected everywhere.
   280Ôćĺ
   281Ôćĺ### 3.4 No internationalization despite ASEAN target
   282Ôćĺ
   283ÔćĺZero i18n. All UI strings hardcoded English. Currency hardcoded "PHP" despite targeting
   284ÔćĺPH/ID/MY/TH/VN (IDR/MYR/THB/VND). Demo phone number is PH format. The chatbot backend claims
   285ÔćĺTagalog/Cebuano/Bahasa/Thai/Vietnamese support but the frontend sends raw text only and there
   286Ôćĺis no language detection.
   287Ôćĺ
   288Ôćĺ**Remediation:** i18next (or equivalent) with locale switching. Per-country currency
   289Ôćĺformatting via `Intl.NumberFormat`. RTL support (not needed for ASEAN but good practice).
   290ÔćĺLocale-negotiated chatbot. Translatable route metadata.
   291Ôćĺ
   292Ôćĺ### 3.5 No PWA / offline support
   293Ôćĺ
   294ÔćĺNo `manifest.json`, no service worker, no Workbox. Map tiles cached only by the browser HTTP
   295Ôćĺcache ÔÇö a hard refresh or first-load offline yields a blank map. No network-status detection;
   296Ôćĺif the browser goes offline, all polling fetches fail silently.
   297Ôćĺ
   298Ôćĺ**Remediation:** PWA with a service worker (Workbox) for app-shell caching, offline route
   299Ôćĺcache, and stale-while-revalidate for map tiles. Network-status detection with a visible
   300Ôćĺ"offline" banner. Install prompt for commuter app.
   301Ôćĺ
   302Ôćĺ### 3.6 No state management, no server-state library
   303Ôćĺ
   304ÔćĺA single mutable `state` object with 40+ properties (`core.js:2-42`) shared across all 8
   305Ôćĺscripts. Any function can mutate any field. Re-rendering is manual (`renderMobile()` /
   306Ôćĺ`renderOperator()`) and easy to forget. The same "vehicle card" is rendered in 5+ places with
   307Ôćĺsubtle differences. Polling uses `setInterval` every 30s (mobile) / 15s (operator) with no
   308Ôćĺbackoff, no pause-on-hidden-tab, no overlap protection. No WebSocket ÔÇö the README claims
   309Ôćĺ`/ws/fleet` exists but the frontend has zero `new WebSocket` references and the backend
   310Ôćĺendpoint is `/ws/telemetry` (not `/ws/fleet`).
   311Ôćĺ
   312Ôćĺ**Remediation:** TanStack Query for server state (caching, polling, retries, dedup, pause on
   313Ôćĺhidden tab). Zustand for client state. React Server Components / Next.js for the shell.
   314ÔćĺWebSocket (socket.io) for live fleet updates with auto-reconnect.
   315Ôćĺ
   316Ôćĺ### 3.7 No dark mode, no design system
   317Ôćĺ
   318ÔćĺZero `prefers-color-scheme` matches. The "Dark" map tile layer only changes map tiles, not
   319ÔćĺUI. `variables.css` defines only 14 CSS custom properties, but ~30 different hex values are
   320Ôćĺhardcoded across JS and CSS *not* in the variable set. Three overlapping modal systems
   321Ôćĺ(`.ls-modal-*`, `.modal`/`.modal-content`, `.action-modal*`). `!important` used 18+ times.
   322Ôćĺ`map.css` defines the same `.map-type-card` rules verbatim in two places (lines 602-645 and
   323Ôćĺ797-844).
   324Ôćĺ
   325Ôćĺ**Remediation:** Tailwind CSS 4 with a full design-token scale (color, spacing, typography,
   326Ôćĺradius, shadow, motion). Dark mode via token swap. shadcn/ui component library for consistent
   327Ôćĺprimitives. `stylelint` + `eslint-plugin-tailwindcss` to enforce token usage.
   328Ôćĺ
   329Ôćĺ### 3.8 Vendored libraries committed uncompressed
   330Ôćĺ
   331Ôćĺ`app/vendor/leaflet/` contains `leaflet.js` (147 KB uncompressed), `leaflet.markercluster.js`
   332Ôćĺ(34 KB), CSS, and PNG marker assets ÔÇö all committed with no version pin, no SRI, no license
   333Ôćĺfile. No minification, no tree-shaking (only a fraction of Leaflet's API is used).
   334Ôćĺ
   335Ôćĺ**Remediation:** npm-managed dependencies with pinned versions and `package-lock.json`. Vite
   336Ôćĺhandles bundling, tree-shaking, and minification. SRI hashes on third-party CDN assets.

   339Ôćĺ
   340Ôćĺ## 4. Edge-layer integrity failures
   341Ôćĺ
   342ÔćĺThis is the most serious category. The original LoadSense **pitched YOLOv8-nano computer
   343Ôćĺvision** but the edge code does not do computer vision.
   344Ôćĺ
   345Ôćĺ### 4.1 `line_crossing_counter.py` `webcam`/`video` modes are fake CV
   346Ôćĺ
   347ÔćĺThe file has three modes via `--source`:
   348Ôćĺ
   349Ôćĺ- `sim` (default): pure random-walk in pixel space. Honest, labeled as simulation.
   350Ôćĺ- `webcam` / `video`: `import cv2`, `cv2.VideoCapture(source)`, reads frames in a loop ÔÇö
   351Ôćĺ  **then ignores every pixel of every frame**. The only frame-derived value is
   352Ôćĺ  `gray_mean = int(frame.mean())` (line 103), and `movement = ((gray_mean % 17) - 8)` (line
   353Ôćĺ  104). **No object detection, no tracking, no YOLO, no DeepSORT/ByteTrack.** Frame brightness
   354Ôćĺ  mod 17 is treated as "person movement".
   355Ôćĺ
   356ÔćĺYet `docs/loadsense_pitch_outline.md` Section 5A explicitly states: *"We run a lightweight
   357Ôćĺcomputer vision modelÔÇöspecifically object detection like YOLOÔÇöto track passengers crossing
   358Ôćĺthe door threshold."* The code does not deliver this; the pitch does. This is an **integrity
   359Ôćĺgap**, not merely a technical debt item.
   360Ôćĺ
   361Ôćĺ**Remediation:** Re-LoadSense is honest. The edge layer is either (a) real YOLOv8-nano via
   362ÔćĺUltralytics running on a Raspberry Pi 5 / Jetson Nano, with ByteTrack for identity tracking,
   363Ôćĺor (b) a clearly-labeled `sim` mode that emits the same contract. There is no `webcam` mode
   364Ôćĺthat opens a camera and ignores pixels. The contract between edge and cloud is a versioned
   365Ôćĺprotobuf/JSON Schema, not ad-hoc.
   366Ôćĺ
   367Ôćĺ### 4.2 No edge-cloud contract schema
   368Ôćĺ
   369ÔćĺThe telemetry payload is ad-hoc JSON with no formal schema, no version field, no device
   370Ôćĺidentity, no firmware version, no auth token, no heartbeat. `mock_telemetry.py` and
   371Ôćĺ`line_crossing_counter.py` emit slightly different field sets. No protobuf, no Avro, no JSON
   372ÔćĺSchema, no AsyncAPI.
   373Ôćĺ
   374Ôćĺ**Remediation:** Versioned edge-cloud contract (protobuf or JSON Schema with a `schema_version`
   375Ôćĺfield). AsyncAPI spec for the telemetry channel. Device identity, firmware version, and auth
   376Ôćĺtoken in every payload. Heartbeat/liveness separate from telemetry.
   377Ôćĺ
   378Ôćĺ### 4.3 No offline buffer, no device management, no OTA
   379Ôćĺ
   380Ôćĺ- HTTP failure on the edge = data loss. `post_telemetry` prints to stderr and drops the row.
   381Ôćĺ- No device registry (which Pi is on which vehicle).
   382Ôćĺ- No device provisioning flow, no device-to-vehicle binding API.
   383Ôćĺ- No OTA firmware updates, no A/B partition, no rollback.
   384Ôćĺ- No edge ML lifecycle ÔÇö no model download from cloud to edge, no versioned edge model, no
   385Ôćĺ  edge inference metric reporting.
   386Ôćĺ- No edge auth ÔÇö no API key, no mTLS, no JWT, no per-device credentials.
   387Ôćĺ- No edge runtime ÔÇö no container image, no systemd unit, no K3s/balena.
   388Ôćĺ
   389Ôćĺ**Remediation:** A real edge platform: device registry + provisioning API, mTLS device
   390Ôćĺcertificates, offline buffer (SQLite/RocksDB queue on device), batched compressed upload with
   391Ôćĺretry/backoff, OTA firmware pipeline with A/B partitions and automatic rollback, edge model
   392Ôćĺdownload with version negotiation, edge inference metrics reported to cloud, containerized
   393Ôćĺedge runtime (balena/balenaCloud or K3s).
   394Ôćĺ
   395Ôćĺ### 4.4 Privacy leakage pattern
   396Ôćĺ
   397ÔćĺThe pitch claims "no video leaves the device" but `line_crossing_counter.py` writes
   398Ôćĺ`centroid_x`, `centroid_y`, `density_zone` per frame to a CSV committed to the repo. This is
   399Ôćĺa metadata leakage pattern that real YOLO bbox data would inherit unless explicitly redacted.
   400Ôćĺ
   401Ôćĺ**Remediation:** Edge devices emit only aggregate counts (boarded, alighted, current
   402Ôćĺoccupancy, density-zone histogram). No per-frame coordinates, no bboxes, no images ever leave
   403Ôćĺthe device. A privacy impact assessment (PIA) is written and reviewed before any pilot.
   404Ôćĺ

   407Ôćĺ## 5. ML / MLOps failures
   408Ôćĺ
   409ÔćĺThe ML layer (`cloud/` + `backend/app/core/phase2.py`) is broken end-to-end.
   410Ôćĺ
   411Ôćĺ### 5.1 Trained artifacts do not exist
   412Ôćĺ
   413ÔćĺREADME, `phase-2.md`, `cloud/README.md`, and `DATA_SOURCES_AND_APIS.md` all claim "checked-in
   414Ôćĺartifacts" but `cloud/artifacts/` **does not exist in the repo**. `phase2.py:load_eta_model()`
   415Ôćĺreturns `None` (line 20) and `predict_eta_details` silently falls back to a config-driven
   416Ôćĺheuristic returning `{"source": "fallback"}`. The trained model is effectively dead code in
   417Ôćĺproduction.
   418Ôćĺ
   419Ôćĺ### 5.2 Train/serve path mismatch for demand forecast
   420Ôćĺ
   421Ôćĺ`train_demand_forecast.py` writes to `cloud/artifacts/demand_forecast.json` (single file).
   422Ôćĺ`phase2.py:60-61` reads `cloud/artifacts/demand/{COUNTRY}_demand_forecast.json` (5 per-country
   423Ôćĺfiles). **They never connect.** Even if you run the trainer successfully, the backend will
   424Ôćĺnever pick up the output.
   425Ôćĺ
   426Ôćĺ### 5.3 Target leakage in ETA training
   427Ôćĺ
   428Ôćĺ`train_eta_model.py:build_target()` (lines 23-25) constructs the target as a deterministic
   429Ôćĺlinear function of three of the four input features:
   430Ôćĺ
   431Ôćĺ```python
   432Ôćĺnoise = (frame["stop_index"] % 3) * 0.2
   433Ôćĺreturn 4.5 + frame["stop_index"] * 0.8 + frame["time_of_day"] * 0.06
   434Ôćĺ            + frame["traffic_factor"] * 1.9 + frame["count"] * 0.08 + noise
   435Ôćĺ```
   436Ôćĺ
   437ÔćĺThe GradientBoostingRegressor trivially memorizes the linear coefficients and reports an MAE
   438Ôćĺnear the noise floor (~0.2 minutes). The reported evaluation is **meaningless**. No held-out
   439Ôćĺroute, no held-out time period, no k-fold CV, no baseline comparison.
   440Ôćĺ
   441Ôćĺ### 5.4 Non-deterministic demand forecast in production
   442Ôćĺ
   443Ôćĺ`phase2.py:118` (the no-artifact fallback) uses `random.uniform(-0.5, 0.5)` without a seed.
   444ÔćĺEvery call to `GET /api/demand` returns **different numbers** for the same route and hour. The
   445Ôćĺ`model` field is `"dynamic_simulation"` but the API gives no indication the values are non-
   446Ôćĺdeterministic. Operators would see the dashboard "jitter" between refreshes.
   447Ôćĺ
   448Ôćĺ### 5.5 `pickle.load` is an RCE vector
   449Ôćĺ
   450Ôćĺ`phase2.py:19` does `pickle.load(handle)` on `cloud/artifacts/eta_model.pkl`. `pickle.load`
   451Ôćĺexecutes arbitrary code from the file. Combined with `CORS allow_origins=["*"]` and no auth,
   452Ôćĺthis is a critical-severity issue: if an attacker can write the `.pkl` (path traversal,
   453Ôćĺsupply-chain compromise, misconfigured CI), they get remote code execution on every `/api/eta`
   454Ôćĺcall.
   455Ôćĺ
   456Ôćĺ### 5.6 Heavy deps for dead code
   457Ôćĺ
   458Ôćĺ`requirements.txt` ships `xgboost>=2.0`, `prophet>=1.1`, `google-genai>=0.3.0`. Grep confirms
   459Ôćĺ**xgboost is never imported** (only sklearn's `GradientBoostingRegressor` is used). `prophet`
   460Ôćĺ(~200 MB with cmdstanpy) is used by a trainer whose output is never wired into production.
   461Ôćĺ`google-genai` is consumed only by dead `chatbot.py`. These bloat the Docker image by hundreds
   462Ôćĺof MB for no benefit.
   463Ôćĺ
   464Ôćĺ### 5.7 No MLOps whatsoever
   465Ôćĺ
   466ÔćĺNo model registry (MLflow/W&B), no experiment tracking, no feature store (train-serve skew is
   467Ôćĺstructurally guaranteed), no data validation (Great Expectations/Pandera), no drift detection
   468Ôćĺ(evidently/alibi-detect), no production monitoring (no Prometheus metrics on `/api/eta` or
   469Ôćĺ`/api/demand`), no A/B testing, no canary, no CI/CD for ML, no model serving infrastructure
   470Ôćĺ(no Triton/BentoML/TorchServe). The "MLOps" is: developer runs a script locally, commits a
   471Ôćĺ`.pkl`, FastAPI loads it via `@lru_cache`.
   472Ôćĺ
   473Ôćĺ### 5.8 Synthetic data is unrealistic and unseeded
   474Ôćĺ
   475Ôćĺ`data/generate_synthetic_history.py` produces 500 rows, 15-min intervals, 4 routes cycled
   476Ôćĺ(only 4 of 74 PH routes), `count` as a linear function of `stop_index` and `traffic_factor`.
   477Ôćĺ`random` is **unseeded** ÔÇö every run produces a different CSV. No day-of-week, no holiday
   478Ôćĺeffects, no weather, no events, no spatial correlation. The committed snapshot is one
   479Ôćĺarbitrary realization.
   480Ôćĺ
   481Ôćĺ**Remediation (all of ┬ž5):** Real MLOps stack ÔÇö MLflow for registry + tracking, Feast (or
   482Ôćĺinline typed feature definitions) for feature parity, evidently for drift detection,
   483ÔćĺPrometheus metrics on inference endpoints, joblib/ONNX for safe model serialization (never
   484Ôćĺpickle for untrusted paths), seeded reproducible data generation, train/serve schema
   485Ôćĺvalidation, champion/challenger promotion pipeline, CI/CD that retrains on data refresh and
   486Ôćĺpromotes models through dev Ôćĺ staging Ôćĺ prod with automated rollback.

   691Ôćĺ[`04-roadmap-and-milestones.md`](./04-roadmap-and-milestones.md) sequences the work so that
   692Ôćĺthe most integrity-critical fixes (honest edge, real auth, reproducible ML) land first.
