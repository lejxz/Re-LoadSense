    53Ôćĺ## 2. Real-time map & ETA
    54Ôćĺ
    55Ôćĺ### C-01 ÔÇö Live fleet map with color-coded occupancy markers ÔÇö P0
    56Ôćĺ
    57Ôćĺ**Explanation.** The map is the home screen of the commuter app. It answers the single most
    58Ôćĺimportant question ÔÇö *is the next jeepney on this route worth boarding?* ÔÇö at a glance, using
    59Ôćĺthe four-tier color taxonomy (­ččó available / ­ččí filling / ­čö┤ at capacity / ­čö┤-blink overloaded)
    60Ôćĺthat survived unchanged from the original LoadSense. Without this, Re-LoadSense has no commuter
    61Ôćĺvalue.
    62Ôćĺ
    63Ôćĺ**Description.** A full-screen Leaflet or MapLibre GL map centered on the commuter's current
    64Ôćĺlocation (or last-known city). PUV markers are color- and label-coded by occupancy tier.
    65ÔćĺMarkers update in real time via a WebSocket subscription scoped to the visible bounding box ÔÇö
    66Ôćĺthe server filters, the client never receives the whole fleet. Tapping a marker opens a detail
    67Ôćĺcard (vehicle ID, route code, ETA at the nearest stop, current tier, last-updated timestamp).
    68ÔćĺWhen the network drops, the map falls back to cached tiles (stale-while-revalidate) and shows
    69Ôćĺan "offline" banner; markers freeze at their last-known position with a stale-timestamp badge.
    70Ôćĺ
    71Ôćĺ**Implementation.**
    72Ôćĺ- Frontend: `apps/commuter/app/(map)/page.tsx` ÔÇö a Next.js App Router page hosting the
    73Ôćĺ  `<FleetMap>` client component.
    74Ôćĺ- Map library: `maplibre-gl` (OSS, no API key, supports vector + raster tiles).
    75Ôćĺ- WebSocket: socket.io-client Ôćĺ `B-05` WebSocket gateway; subscription message carries the
    76Ôćĺ  bbox; see [`../architecture/05-api-design.md` ┬žwebsocket-contract`](../architecture/05-api-design.md).
    77Ôćĺ- Tile caching: Workbox `CacheFirst` for raster tiles, `StaleWhileRevalidate` for vector
    78Ôćĺ  tiles, 7-day TTL, 50MB cap.
    79Ôćĺ- Markers: React components; tier color comes from the design-system token scale
    80Ôćĺ  (`--tier-available`, `--tier-filling`, `--tier-capacity`, `--tier-overloaded`).
    81Ôćĺ- XSS: React auto-escaping; no `dangerouslySetInnerHTML` anywhere; popups are React-rendered.
    82Ôćĺ- See [`../architecture/07-frontend-architecture.md` ┬žreal-time`](../architecture/07-frontend-architecture.md).
    83Ôćĺ
    84Ôćĺ**Acceptance.**
    85Ôćĺ- Map renders within 2s of page load on a mid-range Android phone (Chrome).
    86Ôćĺ- Markers reflect the current occupancy tier with both color **and** an accessible text label
    87Ôćĺ  (never color alone ÔÇö see C-12).
    88Ôćĺ- Markers update within 3s of a telemetry event on the backend.
    89Ôćĺ- Tapping a marker opens a detail card (vehicle ID, route, ETA, tier, last updated).
    90Ôćĺ- Map subscribes only to the visible bounding box (server filters); the client never receives
    91Ôćĺ  vehicles outside the bbox.
    92Ôćĺ- Works offline using cached tiles (stale-while-revalidate) with a visible "offline" banner.
    93Ôćĺ- No XSS vectors (React auto-escaping; no `dangerouslySetInnerHTML`).
    94Ôćĺ
    95Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž3.2 (XSS via direct DOM mutation)` and
    96Ôćĺ`┬ž3.6 (no real-time state management in frontend despite a WS claim)`.
    97Ôćĺ
    98Ôćĺ**Simulation scope.** Ôťů Full. MapLibre GL JS with OSM raster tiles. Bbox subscription over
    99Ôćĺsocket.io. Cached tiles via Workbox. The "live" markers are driven by the honest `sim` mode
   100Ôćĺ(E-11) fleet simulator; every marker carries a small "SIM" badge.

   104Ôćĺ### C-02 ÔÇö Stop-level ETA prediction display ÔÇö P0
   105Ôćĺ
   106Ôćĺ**Explanation.** Knowing a jeepney is "filling" is half the answer; the other half is *when
   107Ôćĺwill it arrive at my stop?* This feature surfaces a real, versioned ETA model ÔÇö not a synthetic
   108Ôćĺformula. It is what separates a real transit app from a coloured map.
   109Ôćĺ
   110Ôćĺ**Description.** For a selected vehicle, the detail card (C-01) and trip planner (C-04) both
   111Ôćĺdisplay a list of remaining stops on the route with a predicted arrival time at each. Each ETA
   112Ôćĺincludes a confidence band (e.g. "3ÔÇô5 min") during cold-start periods when the model has low
   113Ôćĺdata, and a tighter single-value estimate once warm. The ETA refreshes when the vehicle moves
   114Ôćĺor every 30s, whichever is first. If the model is unavailable (degraded mode), the UI shows
   115Ôćĺ"historical average" with an explicit, honest source label ÔÇö never a fabricated prediction.
   116Ôćĺ
   117Ôćĺ**Implementation.**
   118Ôćĺ- Backend: `M-03` ETA model served via `GET /api/v1/eta/{vehicle_id}` returning
   119Ôćĺ  `{stops: [{stop_id, name, eta_min, eta_min_low, eta_min_high, source}], model_version,
   120Ôćĺ  generated_at}`. See [`../architecture/05-api-design.md` ┬žeta-endpoint`](../architecture/05-api-design.md).
   121Ôćĺ- Frontend: `apps/commuter/components/StopEtaList.tsx`; polls the endpoint with SWR
   122Ôćĺ  (`refreshInterval: 30s`, `revalidateOnFocus: false`).
   123Ôćĺ- Cache: ETA response cached 30s in Redis (B-03) Ôćĺ deterministic within the cache window (no
   124Ôćĺ  jitter between refreshes).
   125Ôćĺ- Source field: enum `{ "model_vX.Y" | "historical_mean" | "cold_start" }` rendered with an
   126Ôćĺ  icon and tooltip.
   127Ôćĺ- See [`../architecture/06-ml-pipeline.md` ┬žeta-serving`](../architecture/06-ml-pipeline.md).
   128Ôćĺ
   129Ôćĺ**Acceptance.**
   130Ôćĺ- ETA values are sourced from the versioned ONNX ETA model (`M-03`), not a synthetic formula.
   131Ôćĺ- ETA includes a confidence band (e.g. "3ÔÇô5 min") during cold-start periods.
   132Ôćĺ- ETA refreshes when the vehicle moves or every 30s, whichever is first.
   133Ôćĺ- If the model is unavailable, UI shows "historical average" with an honest source label.
   134Ôćĺ- ETA values are deterministic within a 30s cache window (no jitter between refreshes).
   135Ôćĺ
   136Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž5.3 (target leakage ÔÇö the original
   137Ôćĺ`predict_eta` was defined but never called; Re-LoadSense ships a real model)`.
   138Ôćĺ
   139Ôćĺ**Simulation scope.** Ôťů Full. The sim ships `M-03` trained on simulated telemetry with the
   140Ôćĺsame ONNX export path; in the sim the "historical_mean" fallback is the dominant path during
   141Ôćĺthe first hours until enough data accumulates.

   146Ôćĺ
   147Ôćĺ### C-03 ÔÇö Boarding-assistant chatbot ÔÇö P0
   148Ôćĺ
   149Ôćĺ**Explanation.** Many commuters ÔÇö especially first-time users and tourists ÔÇö find a map
   150Ôćĺoverwhelming and would rather just ask *"which jeepney is least crowded right now?"* in their
   151Ôćĺown language. The chatbot is the conversational front door to the same live fleet state. The
   152Ôćĺoriginal LoadSense shipped **five chatbot files** (two of them dead); this feature consolidates
   153Ôćĺto a single service with a strategy interface.
   154Ôćĺ
   155Ôćĺ**Description.** A single chatbot endpoint (`POST /api/v1/chat`) backed by
   156Ôćĺ`chatbot_service.py` with two interchangeable strategies selected by config/feature flag:
   157Ôćĺ- **Heuristic strategy** ÔÇö always available, fully offline, grounded in live fleet + route
   158Ôćĺ  metadata; never hallucinates route codes or vehicle IDs.
   159Ôćĺ- **LLM strategy** ÔÇö RAG-grounded: cited entities (route codes, vehicle IDs, stop names) must
   160Ôćĺ  exist in retrieved context; the post-processor strips any cited entity not in the context
   161Ôćĺ  window.
   162Ôćĺ
   163ÔćĺThe chatbot detects query language across the 7 ASEAN locales (en, fil, ceb, id, ms, th, vi)
   164Ôćĺand responds in the same language. PII (phone numbers, emails) is redacted before persistence.
   165ÔćĺPer-user rate limit: 10 queries/min. Chat history persists across sessions for authenticated
   166Ôćĺusers. An ARIA live region announces bot responses to screen readers.
   167Ôćĺ
   168Ôćĺ**Implementation.**
   169Ôćĺ- Backend: `services/chatbot_service.py` ÔÇö strategy interface, factory, PII redaction,
   170Ôćĺ  language detection (`langdetect` or `lingua-py`).
   171Ôćĺ- Heuristic strategy: `services/chatbot/heuristic.py` ÔÇö template-based NLG over live fleet
   172Ôćĺ  query results.
   173Ôćĺ- LLM strategy: `services/chatbot/llm.py` ÔÇö RAG over `routes`, `stops`, `vehicles`; LLM
   174Ôćĺ  provider pluggable (OpenAI / Gemini / Ollama); post-processing validator.
   175Ôćĺ- Persistence: `chatbot_queries` table (PII-redacted), TTL 30d per S-12.
   176Ôćĺ- Rate limit: Redis token bucket, 10/min per user (S-04).
   177Ôćĺ- Frontend: `apps/commuter/app/(chat)/page.tsx` with `useChat()` hook; ARIA live region via
   178Ôćĺ  `aria-live="polite"` on the response container.
   179Ôćĺ- See [`../architecture/06-ml-pipeline.md` ┬žchatbot`](../architecture/06-ml-pipeline.md) and
   180Ôćĺ  [`../architecture/05-api-design.md` ┬žchatbot-endpoint`](../architecture/05-api-design.md).
   181Ôćĺ
   182Ôćĺ**Acceptance.**
   183Ôćĺ- Single chatbot service with a strategy interface (heuristic + optional LLM).
   184Ôćĺ- Heuristic strategy is always available offline; never hallucinates route facts.
   185Ôćĺ- LLM strategy uses RAG: cited route codes / vehicle IDs must exist in retrieved context.
   186Ôćĺ- Detects query language (en, fil, ceb, id, ms, th, vi) and responds in the same language.
   187Ôćĺ- PII (phone numbers, email) is redacted before persistence.
   188Ôćĺ- Per-user rate limit (10 queries/min).
   189Ôćĺ- Chat history persists across sessions for authenticated users.
   190Ôćĺ- ARIA live region announces bot responses to screen readers.
   191Ôćĺ
   192Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž2.1 (5 chatbot files, three dead)` and
   193Ôćĺ`┬ž3.4 (no i18n ÔÇö the original chatbot was English-only)`.
   194Ôćĺ
   195Ôćĺ**Simulation scope.** Ôťů Heuristic strategy only in the sim. The LLM strategy is documented
   196Ôćĺand the strategy interface is shipped, but the sim runs the heuristic path exclusively to keep
   197Ôćĺthe Vercel build hermetic (no external LLM API keys required). RAG post-processor unit tests
   198Ôćĺstill run in CI.

   202Ôćĺ### C-04 ÔÇö Multi-leg trip planning ÔÇö P1
   203Ôćĺ
   204Ôćĺ**Explanation.** Most ASEAN transit journeys are multi-leg: walk Ôćĺ board jeepney A Ôćĺ transfer
   205ÔćĺÔćĺ board jeepney B Ôćĺ walk. A single-leg "nearest vehicle" answer is not enough. This feature
   206Ôćĺplans multi-leg journeys with live occupancy and ETA per leg, so a commuter can choose a less
   207Ôćĺcrowded route even if it has one more transfer.
   208Ôćĺ
   209Ôćĺ**Description.** Given an origin and destination (lat/lon or place IDs from C-06), the planner
   210Ôćĺreturns at least three ranked journey suggestions. Each suggestion shows: total time, total
   211Ôćĺwalking distance, board/alight stops per leg, live occupancy tier for the boarding vehicle, and
   212ÔćĺETA per leg. The planner respects a configurable maximum walking distance (default 500m,
   213Ôćĺrelaxed to 1500m when no shorter route exists). If no route exists, the UI shows an explanatory
   214Ôćĺmessage ÔÇö never an empty state. Results update if the underlying fleet state changes
   215Ôćĺsignificantly (debounced 5s).
   216Ôćĺ
   217Ôćĺ**Implementation.**
   218Ôćĺ- Backend: `services/trip_planner.py` (the original `transit.py` god-module was split per
   219Ôćĺ  `../legacy-analysis/lessons-learned.md ┬ž2.5`; trip planning is one of the 6 resulting
   220Ôćĺ  modules).
   221Ôćĺ- Algorithm: graph search over the route-stop bipartite graph (Neo4j-style traversal in
   222Ôćĺ  Postgres using `WITH RECURSIVE`); edges weighted by live ETA + walking time + a small
   223Ôćĺ  transfer penalty.
   224Ôćĺ- Live occupancy: joined from Redis live state (B-03).
   225Ôćĺ- Walking distance: `PostGIS ST_Distance` on the routes' LineString geometry.
   226Ôćĺ- Endpoint: `POST /api/v1/trip-suggestions`.
   227Ôćĺ- Frontend: `apps/commuter/app/(plan)/page.tsx`; results render as a vertical timeline.
   228Ôćĺ- See [`../architecture/05-api-design.md` ┬žtrip-suggestions`](../architecture/05-api-design.md).
   229Ôćĺ
   230Ôćĺ**Acceptance.**
   231Ôćĺ- Returns at least 3 ranked suggestions when a single-leg route exists.
   232Ôćĺ- Each suggestion shows: total time, walking distance, board/alight stops, live occupancy
   233Ôćĺ  tier, ETA for each leg.
   234Ôćĺ- Handles "no route found" gracefully with an explanatory message.
   235Ôćĺ- Respects a maximum walking distance (configurable, default 500m, relaxed 1500m).
   236Ôćĺ- Results update if the underlying fleet state changes significantly (debounced).
   237Ôćĺ
   238Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž2.5 (god-module ÔÇö transit.py was 1,860
   239Ôćĺlines mixing routing, geocoding, ETA, demand, simulation, and config)`.
   240Ôćĺ
   241Ôćĺ**Simulation scope.** Ôťů Full. Same algorithm in the sim, run against the seeded route graph.
   242ÔćĺThree cities seeded: Cebu (PH), Jakarta (ID), Bangkok (TH) ÔÇö one per locale cluster.

   246Ôćĺ### C-05 ÔÇö Route directory & search ÔÇö P1
   247Ôćĺ
   248Ôćĺ**Explanation.** Power commuters and tourists want to browse all routes in their city, filter
   249Ôćĺby region, and see which routes currently have live vehicles. The original LoadSense returned
   250Ôćĺthe entire route list with no pagination, no search, and no filtering ÔÇö unusable at scale.
   251Ôćĺ
   252Ôćĺ**Description.** A searchable, filterable, paginated list of routes. Each row shows: route
   253Ôćĺcode, name, region, live vehicle count, and a "view on map" action. Filtering by country,
   254Ôćĺregion, and "has live vehicles"; full-text search on route code and name. Cursor-based
   255Ôćĺpagination (default 20, max 100). Tapping a route shows its geometry on the map and its live
   256Ôćĺvehicles. Searchable dropdowns are accessible (ARIA listbox, keyboard arrow nav).
   257Ôćĺ
   258Ôćĺ**Implementation.**
   259Ôćĺ- Backend: `GET /api/v1/routes?cursor=&limit=&q=&country=&region=&has_live=` Ôćĺ cursor-paginated
   260Ôćĺ  list with a `next_cursor` field; full-text search on a `tsvector` index over `code || name`.
   261Ôćĺ- Live vehicle count: a `LEFT JOIN` against the Redis-backed live state (B-03) materialized as
   262Ôćĺ  a Postgres view refreshed every 30s.
   263Ôćĺ- Frontend: `apps/commuter/app/(routes)/page.tsx` with `@tanstack/react-virtual` for
   264Ôćĺ  row virtualization on long lists; `cmdk` for the searchable dropdown.
   265Ôćĺ- See [`../architecture/05-api-design.md` ┬žpagination`](../architecture/05-api-design.md) and
   266Ôćĺ  [`../architecture/04-data-model.md` ┬žroutes`](../architecture/04-data-model.md).
   267Ôćĺ
   268Ôćĺ**Acceptance.**
   269Ôćĺ- Cursor-based pagination (default 20 per page, max 100).
   270Ôćĺ- Filter by country, region, and "has live vehicles."
   271Ôćĺ- Full-text search on route code and name.
   272Ôćĺ- Tapping a route shows its geometry on the map and its live vehicles.
   273Ôćĺ- Searchable dropdowns are accessible (ARIA listbox, keyboard arrow nav).

   282Ôćĺ### C-06 ÔÇö Place search & geocoding ÔÇö P1
   283Ôćĺ
   284Ôćĺ**Explanation.** Trip planning (C-04) needs an origin and destination. Most commuters don't
   285Ôćĺtype lat/lon ÔÇö they type "Ayala Center Cebu" or "Barangay Lahug." This feature provides the
   286Ôćĺgeocoder front-end, with rate-limiting and graceful degradation that the original lacked.
   287Ôćĺ
   288Ôćĺ**Description.** A debounced search box that calls a geocoder (Photon / Nominatim) and returns
   289Ôćĺranked place suggestions. Results are debounced (300ms), cached (TTL 5min, bounded LRU max
   290Ôćĺ1000 entries), and rate-limited (30 req/min per user). The geocoder is called with a valid
   291Ôćĺcontact User-Agent per Nominatim usage policy. If the geocoder is unavailable, the UI shows
   292Ôćĺ"search unavailable" ÔÇö never a silent failure. No PII in search queries is persisted.
   293Ôćĺ
   294Ôćĺ**Implementation.**
   295Ôćĺ- Backend: `GET /api/v1/places?q=&lat=&lon=&limit=` proxy to Photon
   296Ôćĺ  (`https://photon.komoot.io/api/?q=...`) with our server-side User-Agent.
   297Ôćĺ- Cache: Redis, key `places:{q}:{lat-rounded}:{lon-rounded}`, TTL 5min, LRU eviction at 1000
   298Ôćĺ  entries per worker.
   299Ôćĺ- Rate limit: Redis token bucket, 30/min per user (S-04).
   300Ôćĺ- Frontend: `apps/commuter/components/PlaceSearch.tsx` using `cmdk` + debounce.
   301Ôćĺ- See [`../architecture/05-api-design.md` ┬žplaces-endpoint`](../architecture/05-api-design.md).
   302Ôćĺ
   303Ôćĺ**Acceptance.**
   304Ôćĺ- Uses Photon geocoder with a valid contact User-Agent (Nominatim policy compliance).
   305Ôćĺ- Results are debounced (300ms), cached (TTL 5min, bounded LRU max 1000 entries).
   306Ôćĺ- Rate-limited (30 requests/min per user).
   307Ôćĺ- Graceful degradation: if geocoder is unavailable, show "search unavailable" (no silent
   308Ôćĺ  failure).
   309Ôćĺ- No PII in search queries is persisted.
   310Ôćĺ
   311Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž2.10 (no rate limit ÔÇö the original

   320Ôćĺ## 4. Account & preferences
   321Ôćĺ
   322Ôćĺ### C-07 ÔÇö Commuter account & preferences ÔÇö P1
   323Ôćĺ
   324Ôćĺ**Explanation.** Personalization (saved routes, language, notification opt-in) and data-subject
   325Ôćĺrights (S-13) require an account. The original LoadSense had **zero authentication** ÔÇö anyone
   326Ôćĺcould wipe the DB. This feature ships the commuter-side auth and preferences surface.
   327Ôćĺ
   328Ôćĺ**Description.** Commuters register via email+password (bcrypt) or phone+OTP. Login issues a
   329ÔćĺJWT access token (15min) + refresh token (7d, rotated on each use), both in HttpOnly
   330ÔćĺSameSite=Strict Secure cookies. Preferences include: default country/city, language, theme
   331Ôćĺ(light/dark/system), and notification opt-in. Account deletion (right to erasure, S-13) works
   332Ôćĺwithin 24h.
   333Ôćĺ
   334Ôćĺ**Implementation.**
   335Ôćĺ- Backend: `services/auth_service.py` ÔÇö JWT minting, refresh rotation, revocation list in
   336Ôćĺ  Redis (S-01).
   337Ôćĺ- Password hashing: `bcrypt` with cost 12.
   338Ôćĺ- Phone OTP: integrated with Twilio (or Vercel-compatible SMS provider); OTP stored in Redis
   339Ôćĺ  with 5min TTL, 3-attempt lockout.
   340Ôćĺ- Cookies: `HttpOnly; Secure; SameSite=Strict`; CSRF token double-submit on mutations.
   341Ôćĺ- Frontend: `apps/commuter/app/(auth)/login/page.tsx`, `register/page.tsx`,
   342Ôćĺ  `settings/page.tsx`; middleware reads the access cookie and redirects unauthenticated users.
   343Ôćĺ- See [`../architecture/08-security-architecture.md` ┬žauthn`](../architecture/08-security-architecture.md).
   344Ôćĺ
   345Ôćĺ**Acceptance.**
   346Ôćĺ- Registration via email+password (bcrypt) or phone+OTP.
   347Ôćĺ- JWT access (15min) + refresh (7d, rotated) tokens in HttpOnly SameSite=Strict Secure cookies.
   348Ôćĺ- Preferences: default country/city, language, notification opt-in.
   349Ôćĺ- Account deletion (right to erasure) works within 24h.
   350Ôćĺ
   351Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž2.2 (zero authentication ÔÇö the original
   352Ôćĺallowed `POST /api/database/reset` with no auth)`.
   353Ôćĺ
   354Ôćĺ**Simulation scope.** Ôťů Full. Email+password and phone OTP both work in the sim (OTP sent via
   355Ôćĺconsole log + Vercel email integration). The session cookie pattern works on Vercel's

   360Ôćĺ### C-08 ÔÇö Occupancy notifications ÔÇö P2
   361Ôćĺ
   362Ôćĺ**Explanation.** A commuter waiting at a stop wants to be notified *only* when a vehicle they
   363Ôćĺcare about (right route, right tier) is N minutes away ÔÇö not stare at a map. This is a power
   364Ôćĺfeature; selected for the sim because it's cheap (in-app).
   365Ôćĺ
   366Ôćĺ**Description.** A commuter subscribes to a (route, stop, tier, minutes-away) tuple. When a
   367Ôćĺvehicle matching the subscription is within `minutes-away` of the stop with `tier` ÔëĄ requested,
   368Ôćĺa push notification fires. The notification includes vehicle ID, route, ETA, and tier. Web Push
   369ÔćĺAPI with a service worker. Configurable threshold. Max 5 active subscriptions per commuter
   370Ôćĺ(rate-limit protection). Respects Do Not Disturb hours (configurable).
   371Ôćĺ
   372Ôćĺ**Implementation.**
   373Ôćĺ- Backend: `services/notification_service.py` ÔÇö evaluates subscriptions on each telemetry
   374Ôćĺ  event; matches trigger a notification job.
   375Ôćĺ- Web Push: `web-push` npm package; VAPID keys in env; subscription endpoints stored per user
   376Ôćĺ  in `push_subscriptions` table.
   377Ôćĺ- Service worker: `apps/commuter/public/sw.js` (registered alongside the offline SW in C-09).
   378Ôćĺ- Endpoint: `POST /api/v1/notifications/subscriptions`, `DELETE` for unsubscribe, `GET` to list.
   379Ôćĺ- DND: a `dnd_start` / `dnd_end` local-time pair per user; checks in the matcher.
   380Ôćĺ- See [`../architecture/05-api-design.md` ┬žnotifications`](../architecture/05-api-design.md).
   381Ôćĺ
   382Ôćĺ**Acceptance.**
   383Ôćĺ- Web Push API (with service worker).
   384Ôćĺ- Configurable threshold (tier + minutes away).
   385Ôćĺ- Max 5 active subscriptions per commuter (rate-limit protection).
   386Ôćĺ- Notification includes vehicle ID, route, ETA, tier.
   387Ôćĺ- Respects Do Not Disturb hours (configurable).
   388Ôćĺ
   389Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž3.6 (no real-time ÔÇö the original frontend
   390Ôćĺpolled every 5s with no notification surface)`.
   391Ôćĺ
   392Ôćĺ**Simulation scope.** ­ččí Partial. The sim ships **in-app notifications only** (toast + a
   393Ôćĺnotifications center page). Web Push requires a VAPID key + a registered service worker and is
   394Ôćĺdocumented but gated behind a `NEXT_PUBLIC_ENABLE_WEB_PUSH` flag (off by default in the sim
   395Ôćĺbuild) to keep the Vercel preview deploys hermetic. The matcher, subscription store, and DND
   396Ôćĺlogic all ship.

   400Ôćĺ## 5. PWA & offline
   401Ôćĺ
   402Ôćĺ### C-09 ÔÇö Offline mode & PWA install ÔÇö P1
   403Ôćĺ
   404Ôćĺ**Explanation.** ASEAN commuters frequently lose connectivity mid-journey (tunnels, dead
   405Ôćĺspots, data-capped plans). The app must be installable and usable offline ÔÇö the original was
   406Ôćĺneither. This is what makes Re-LoadSense a *PWA* and not just a website.
   407Ôćĺ
   408Ôćĺ**Description.** The commuter app ships a `manifest.json` (icons, name, theme color,
   409Ôćĺ`start_url`, `display: standalone`) and a Workbox-powered service worker. The SW precaches the
   410Ôćĺapp shell, uses stale-while-revalidate for map tiles, and network-first for fleet data with a
   411Ôćĺcache fallback. An offline banner appears when the network is lost and disappears on reconnect.
   412ÔćĺInstall prompt works on Android Chrome and iOS Safari. Offline-cached routes survive a hard
   413Ôćĺrefresh.
   414Ôćĺ
   415Ôćĺ**Implementation.**
   416Ôćĺ- Frontend: `apps/commuter/public/manifest.webmanifest`; `apps/commuter/public/sw.js`
   417Ôćĺ  generated by `workbox-build` at build time.
   418Ôćĺ- Workbox strategies: `precache` for app shell (`/_next/static/*`, `/icons/*`, route HTML
   419Ôćĺ  shells); `StaleWhileRevalidate` for raster tiles; `NetworkFirst` with `NetworkOnly` fallback
   420Ôćĺ  to cache for `/api/v1/fleet/*` and `/api/v1/eta/*`.
   421Ôćĺ- Offline banner: a React context subscribed to `navigator.onLine` + SW sync events.
   422Ôćĺ- Install prompt: `beforeinstallprompt` event captured and surfaced as an "Install app" button
   423Ôćĺ  in settings; iOS Safari shows a custom "Add to Home Screen" tooltip.
   424Ôćĺ- See [`../architecture/07-frontend-architecture.md` ┬žpwa`](../architecture/07-frontend-architecture.md).
   425Ôćĺ
   426Ôćĺ**Acceptance.**
   427Ôćĺ- `manifest.json` with icons, name, theme color, start_url, display: standalone.
   428Ôćĺ- Service worker (Workbox): app-shell precache, stale-while-revalidate for map tiles,
   429Ôćĺ  network-first for fleet data with cache fallback.
   430Ôćĺ- Offline banner appears when network is lost; disappears on reconnect.
   431Ôćĺ- Install prompt works on Android Chrome and iOS Safari.
   432Ôćĺ- Offline-cached routes survive a hard refresh.
   433Ôćĺ
   434Ôćĺ**Lesson.** `../legacy-analysis/lessons-learned.md ┬ž3.5 (no PWA ÔÇö the original was a plain
   435ÔćĺSPA with no service worker and no install prompt)`.
   436Ôćĺ
   437Ôćĺ**Simulation scope.** Ôťů Full. PWA install works on Vercel preview deploys; the SW is

   518Ôćĺ### C-12 ÔÇö Accessibility (a11y) ÔÇö P0

   542Ôćĺ  capacity`, `Overloaded`) and an icon (circle, half-circle, full-circle, blink).
