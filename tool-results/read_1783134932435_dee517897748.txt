    30→## 1. Executive summary
    31→
    32→### 1.1 The problem (unchanged from the original — it was correct)
    33→
    34→Cebu's commuters, operators, and regulators all operate blind on the one variable that
    35→matters most: **how full is the next jeepney?** Commuters wait 20+ minutes for rides that may
    36→already be full. Drivers exceed legal capacity (*sabit* overloading) because they are paid
    37→per head and there is no live capacity signal. Operators allocate fleets by gut-feel. LGUs
    38→have no city-wide compliance dashboard. Traffic congestion costs the Philippine economy PHP
    39→3.5 billion daily. Google Maps shows *where* a jeepney is; none of them show *whether you can
    40→fit inside it*.
    41→
    42→### 1.2 The solution (rebuilt for production)
    43→
    44→**Re-LoadSense** is a three-tier platform that turns every PUV into a real-time occupancy
    45→sensor:
    46→
    47→- **Tier 1 — Edge (in-vehicle).** An overhead camera runs YOLOv8-nano **fully offline**,
    48→  performing bidirectional passenger counting with ByteTrack identity tracking. The current
    49→  count maps to a four-tier occupancy state and drives a **windshield LED strip**. The edge
    50→  device buffers telemetry offline, authenticates to the cloud with mTLS, and receives OTA
    51→  model + firmware updates.
    52→
    53→- **Tier 2 — Cloud (intelligence & orchestration).** A service-oriented FastAPI backend on
    54→  PostgreSQL + Redis ingests authenticated telemetry, predicts ETA, detects route deviations
    55→  and driving anomalies, forecasts demand, fires operator-first safety alerts, and powers a
    56→  context-aware boarding assistant. Every endpoint is versioned, validated, rate-limited,
    57→  and observable.
    58→
    59→- **Tier 3 — Clients (commuter, operator, regulator).** A Next.js PWA for commuters (offline-
    60→  capable, installable, localized), a Next.js operator console (fleet, demand, alerts,
    61→  verification workflow), and a regulator dashboard (city-wide compliance, aggregated and
    62→  anonymized).
    63→
    64→### 1.3 What changed from the original, and why
    65→
    66→The original LoadSense had the right mission and the right high-level architecture. What it
    67→lacked was engineering rigor. Re-LoadSense preserves the mission and the four-tier occupancy
    68→taxonomy and the operator-first alert workflow — and rebuilds everything else:
    69→
    70→| Dimension | Original LoadSense | Re-LoadSense |
    71→|---|---|---|
    72→| Edge CV | Fake (`frame.mean() % 17`) | Real YOLOv8-nano + ByteTrack, or honestly-labeled `sim` |
    73→| Backend | FastAPI + 5 SQLite files, no auth | FastAPI + PostgreSQL + Redis, JWT/RBAC, mTLS for edge |
    74→| Architecture | Routes with inline business logic; 1,860-line god-module | Layered: routes → services → repositories; split modules |
    75→| ML | pickle.load, target leakage, train/serve mismatch, no MLOps | MLflow registry, Feast feature parity, evidently drift, ONNX serving |
    76→| Frontend | Vanilla HTML/CSS/JS, no build, no a11y, no i18n | Next.js + TypeScript + Tailwind + shadcn/ui, PWA, i18n, a11y |
    77→| Security | CORS `*`, no auth, no rate limit, no headers | Authn/authz, RBAC, rate limit, CSP, SAST/DAST, threat model |
    78→| Observability | `print()`, no metrics, no traces | structlog + Prometheus + OTel + Sentry + Grafana |
    79→| Testing | 3 real pytest assertions, no CI | pytest + Playwright + k6, ≥80% coverage gate, CI on every PR |
    80→| Docs | 9+ ghost citations, drift | Generated from code; CI docs-drift check; ADRs; model cards |
    81→| Operability | Single-stage Docker, root, no backups, no runbooks | Multi-stage non-root, Helm/k8s, backups, runbooks, SLOs |
    82→
    83→### 1.4 Target users
    84→
    85→| User | Primary need | Re-LoadSense surface |
    86→|---|---|---|
    87→| **Commuter** | "Is the next jeepney full? When does it arrive?" | Commuter PWA: live map, ETA, occupancy tier, boarding assistant |
    88→| **PUV operator** | "Where is my fleet? Which routes need more vehicles?" | Operator console: live fleet, demand forecast, alert verification |
    89→| **Driver** | "Am I overloaded? Is my driving flagged?" | Windshield LED + driver-facing minimal display |
    90→| **LGU / regulator** | "Is the city compliant? Where are hotspots?" | Regulator dashboard: aggregated, anonymized compliance metrics |
    91→| **Edge device** | "Securely report counts; get updates" | Edge agent: mTLS auth, offline buffer, OTA, model download |

    95→## 2. Design principles
    96→
    97→These principles govern every decision below. They are derived from the lessons-learned audit.
    98→
    99→1. **Honest scope.** Simulated capabilities are labeled as simulated — in code, docs, and
   100→   pitch. No `webcam` mode that ignores pixels. No "checked-in artifacts" that don't exist.
   101→2. **One source of truth.** This concept is canonical. Code, tests, and docs stay in sync via
   102→   CI; doc drift fails the build. API reference is generated from OpenAPI, not hand-written.
   103→3. **Security by default.** Authn/authz, input validation, rate limiting, and secret
   104→   management land in the first vertical slice — not "phase 2."
   105→4. **Reproducibility over cleverness.** Seeded RNG, pinned dependencies, versioned models,
   106→   hermetic tests. If a build or training run can't be reproduced, it didn't happen.
   107→5. **Operability is a feature.** Structured logs, metrics, traces, health checks, runbooks,
   108→   and rollback procedures ship with the code that needs them.
   109→6. **ASEAN-first, not PH-only.** i18n, multi-currency, per-country data isolation, and
   110→   culturally neutral seed data from day one.
   111→7. **Edge honesty.** The edge layer is either real (YOLOv8-nano on a real device) or a
   112→   contract-faithful simulator with a versioned schema — never a fake.
   113→8. **Layered architecture.** Routes → services → repositories → models. No god-modules. No
   114→   inline business logic in handlers.
   115→9. **Versioned contracts.** API versioning (`/v1/`), edge-cloud schema versioning, model
   116→   versioning, feature schema versioning. Breaking changes are deliberate, not accidental.
   117→10. **Pilotable, not theoretical.** Every component must be deployable in a real Cebu pilot
   118→    with real jeepneys within the roadmap's horizon. No component exists only on paper.

   215→### 3.2 Request flow (example: edge telemetry ingest)
   216→
   217→1. Edge agent captures a frame, runs YOLOv8-nano inference, ByteTrack updates track IDs, the
   218→   line-crossing counter increments board/alight counts, the occupancy tier is computed.
   219→2. The agent batches the telemetry (count delta + GPS + tier + timestamp + device ID +
   220→   firmware version + schema version) into a compressed payload.
   221→3. The agent posts to the edge-ingest endpoint over mTLS. If offline, it buffers to a local
   222→   SQLite queue; a background retry loop drains the queue with exponential backoff.
   223→4. The edge-ingest service validates the mTLS client certificate, verifies the device is
   224→   registered and bound to a vehicle, validates the payload against the versioned schema,
   225→   enriches it (route matching, geofence check), and writes to PostgreSQL + publishes to
   226→   Redis.
   227→5. The FleetService updates the live vehicle state in Redis. The AlertService evaluates
   228→   deviation/anomaly/overload rules and raises alerts (persisted to PostgreSQL).
   229→6. The WebSocket gateway pushes the updated vehicle state to subscribed commuter/operator
   230→   clients.
   231→7. The ETAService may recompute ETA for the vehicle's remaining stops (cached in Redis).
   232→8. Structured logs, metrics, and traces are emitted at each step.
   233→
   234→### 3.3 Multi-tenancy model
   235→
   236→Re-LoadSense is multi-tenant by **country** (and optionally by **operator** within a
   237→country). Unlike the original's file-per-country SQLite fan-out, Re-LoadSense uses a single
   238→PostgreSQL database with:
   239→
   240→- A `country_code` column on every tenant-scoped table.
   241→- A `operator_id` column on operator-scoped tables (vehicles, alerts).
   242→- Row-Level Security (RLS) policies so a regulator in country PH cannot read country TH data
   243→  even if the application layer has a bug.
   244→- Per-country configuration overrides (fare tables, capacity defaults, language, currency)
   245→  stored in a `country_config` table.

   249→## 4. Edge layer
   250→
   251→The edge layer is where the original LoadSense had its most serious integrity failure. Re-
   252→LoadSense treats the edge as a first-class engineered subsystem, not a demo prop.
   253→
   254→### 4.1 Hardware target
   255→
   256→- **Compute:** Raspberry Pi 5 (8 GB) as the baseline; NVIDIA Jetson Nano Orin as the
   257→  high-performance option for higher-framerate inference.
   258→- **Camera:** overhead CSI or USB camera, 720p @ 15 fps minimum. Wide-angle lens for full
   259→  door coverage.
   260→- **GPS:** serial NMEA module (u-blox NEO-6M or equivalent) with external antenna. Cell-tower
   261→  fallback via the modem.
   262→- **Connectivity:** 4G LTE modem (Quectel EC25) for cloud telemetry. The device must operate
   263→  fully offline; connectivity is only for telemetry upload and OTA.
   264→- **LED strip:** WS2812B addressable LED strip driven via GPIO (PWM) for the four-tier
   265→  display (green / yellow / red / blinking-red) visible through the windshield.
   266→- **Power:** 12V→5V buck converter from the vehicle electrical system; 18650 Li-ion buffer
   267→  for graceful shutdown on power loss.
   268→- **Enclosure:** IP54-rated, thermally managed for tropical climate (Cebu ambient up to
   269→  38 °C).
   270→
   271→### 4.2 Edge software stack
   272→
   273→The edge agent runs as a container (Docker / balena) on the device:
   274→
   275→```
   276→edge-agent/
   277→├── camera/              # Camera capture (OpenCV / Picamera2)
   278→├── inference/           # YOLOv8-nano wrapper (Ultralytics)
   279→├── tracking/            # ByteTrack wrapper
   280→├── counter/             # Bidirectional line-crossing counter
   281→├── occupancy/           # Tier calc + hysteresis
   282→├── telemetry/           # Buffer + batch + compress + mTLS upload
   283→├── led/                 # GPIO LED driver
   284→├── ota/                 # Firmware + model update manager (A/B partitions)
   285→├── health/              # Heartbeat + self-diagnostics
   286→├── config/              # Device-local config (device ID, cert paths, endpoints)
   287→└── main.py              # Orchestrator
   288→```
   289→
   290→### 4.3 The counting pipeline (real CV, not fake)
   291→
   292→1. **Capture:** grab a frame at ~10 fps (configurable). Drop frames if inference is slower
   293→   than capture rate (never block).
   294→2. **Detect:** YOLOv8-nano inference. The model is fine-tuned on a labeled dataset of
   295→   overhead PUV door footage (collected under written operator consent during pilot). Output:
   296→   bounding boxes with `person` class confidence.
   297→3. **Track:** ByteTrack assigns persistent track IDs to detected persons across frames. This
   298→   is the piece the original entirely lacked (`person_id` was hardcoded to `1`).
   299→4. **Count:** A virtual line is drawn across the door threshold. A track crossing the line
   300→   in the boarding direction increments `boarded`; the opposite direction increments
   301→   `alighted`. Direction is determined by the track's centroid velocity vector relative to
   302→   the line normal.
   303→5. **Reconcile:** `current_occupancy = max(0, previous_occupancy + boarded - alighted)`.
   304→   Hysteresis prevents flicker between tiers (a vehicle must stay in a new tier for N seconds
   305→   before the tier changes).
   306→6. **Emit:** a telemetry event is enqueued with `{boarded, alighted, current_occupancy, tier,
   307→   gps, timestamp, device_id, firmware_version, model_version, schema_version}`. **No
   308→   images, no bounding boxes, no coordinates leave the device.** Only aggregate counts.
   309→
   310→### 4.4 The `sim` mode (honestly labeled)
   311→
   312→For development, demos, and CI without hardware, the edge agent has a `sim` mode that:
   313→- Generates synthetic frame data from a seeded RNG (reproducible).
   314→- Runs the same counter pipeline (so the downstream contract is identical).
   315→- Emits the same telemetry schema.
   316→- Is labeled `source: "simulator"` in every emitted payload and in every UI that consumes it.
   317→
   318→There is **no** `webcam` mode that opens a camera and ignores pixels. The original's integrity
   319→gap is closed.
   320→
   321→### 4.5 Edge-cloud contract
   322→
   323→The telemetry payload is a **versioned JSON Schema** (with a protobuf definition for
   324→bandwidth-constrained deployments). Every payload includes:
   325→
   326→| Field | Type | Purpose |
   327→|---|---|---|
   328→| `schema_version` | string (semver) | Contract version; enables backward-compatible evolution |
   329→| `device_id` | string (UUID) | Stable device identity (from device registry) |
   330→| `vehicle_id` | string | Bound vehicle (from device-vehicle binding) |
   331→| `firmware_version` | string (semver) | For OTA compatibility checks |
   332→| `model_version` | string (semver) | YOLO model version that produced the counts |
   333→| `timestamp` | ISO 8601 UTC | Event time (device clock, drift-corrected) |
   334→| `gps` | `{lat, lon, accuracy_m, speed_kph, heading}` | Position |
   335→| `counts` | `{boarded, alighted, current_occupancy}` | Aggregate counts only |
   336→| `tier` | enum `available\|filling\|at_capacity\|overloaded` | 4-tier state |
   337→| `signal_quality` | enum | GPS quality indicator |
   338→| `seq` | int | Monotonic sequence number for dedup / gap detection |
   339→
   340→An **AsyncAPI** spec documents the telemetry channel, the heartbeat channel, and the OTA
   341→command channel. The cloud-ingest service validates every payload against the schema;
   342→schema violations are rejected with a structured error and logged.
   343→
   344→### 4.6 Offline resilience
   345→
   346→- **Local buffer:** a SQLite queue on the device stores unsent telemetry. Capacity: ≥7 days
   347→  at peak rate (~1 event/3s/vehicle = ~20k events/day).
   348→- **Batching:** telemetry is batched (e.g., 50 events or 10 seconds, whichever first) and
   349→  gzip-compressed before upload.
   350→- **Retry:** exponential backoff with jitter; max 30s between retries; never drops data
   351→  (unlike the original's silent-drop-on-error).
   352→- **Dedup:** the `seq` number allows the cloud to dedup retransmissions.
   353→- **Clock drift:** the device syncs via NTP when online; offline, the drift is bounded and
   354→  corrected on ingest.
   355→
   356→### 4.7 Device lifecycle management
   357→
   358→- **Registry:** a `devices` table records every provisioned device (ID, certificate serial,
   359→  hardware model, firmware version, bound vehicle, operator, status).
   360→- **Provisioning:** a device is provisioned via a signed enrollment flow that installs its
   361→  mTLS client certificate and initial config. No device can telemeter without a valid cert.
   362→- **Binding:** a device is bound to a vehicle via the operator console; binding changes are
   363→  audited.
   364→- **OTA:** firmware and model updates are delivered via an A/B partition scheme. The new
   365→  image is downloaded, verified (signature), written to the inactive partition, and booted.
   366→  A health check watchdog auto-rollbacks to the previous partition if the new image fails to
   367→  heartbeat within 60s.
   368→- **Decommissioning:** a device can be remotely decommissioned (cert revoked), which blocks
   369→  all future telemetry from it.

   374→
   375→### 5.1 Service-oriented architecture

   377→The backend is a FastAPI application organized into **layered services**:
   378→
   379→```

   385→│   │       ├── routes_telemetry.py
   386→│   │       ├── routes_fleet.py
   387→│   │       ├── routes_routes.py
   388→│   │       ├── routes_alerts.py
   389→│   │       ├── routes_eta.py
   390→│   │       ├── routes_demand.py

   434→handlers contain no business logic. Services contain no SQL. Repositories contain no business
   435→rules. This is the discipline the original lacked.

   437→### 5.2 The `transit` module split
   438→
   439→The original's 1,860-line `transit.py` god-module is split into:
   440→
   441→| Module | Responsibility |
   442→|---|---|
   443→| `services/geocoding_service.py` | Place search, forward/reverse geocoding, Photon/OSM integration with caching |
   444→| `services/route_matching_service.py` | Matching a GPS point to a known route, fuzzy route name matching |
   445→| `services/multi_leg_service.py` | Multi-leg journey planning (origin → walk → board → transfer → alight → walk → destination) |
   446→| `services/fare_service.py` | Per-country fare calculation (config-driven, not hardcoded) |
   447→| `services/nlp_service.py` | Natural-language query parsing for the chatbot (intent + entity extraction) |
   448→| `services/i18n_service.py` | Language detection, translation, locale-aware formatting |
   449→
   450→Each module is independently unit-testable and has a clear interface.
   451→
   452→### 5.3 Live fleet state
   453→
   454→The original held all vehicle state in a process-local `FleetStore` singleton — impossible to
   455→scale across workers. Re-LoadSense holds live state in **Redis** (the source of truth for
   456→"where is vehicle X right now") and persists durable data to **PostgreSQL** (telemetry logs,
   457→alerts, audit). This means:
   458→
   459→- Multiple FastAPI workers (and multiple replicas) share the same live state.
   460→- The WebSocket gateway is a separate service (or a socket.io adapter backed by Redis
   461→  pub/sub) so any worker can push to any subscriber.
   462→- Live reads (`GET /fleet`) hit Redis (sub-millisecond); durable reads hit PostgreSQL.
   463→
   464→### 5.4 Configuration
   465→
   466→A typed configuration system (pydantic-settings) replaces the original's `lru_cache` on a
   467→JSON file:
   468→
   469→```python
   470→class Settings(BaseSettings):
   471→    # Environment
   472→    env: Literal["dev", "staging", "prod"] = "dev"
   473→    demo_mode: bool = False  # DEFAULTS TO FALSE (original defaulted to true)
   474→
   475→    # Database
   476→    database_url: SecretStr
   477→    redis_url: SecretStr
   478→
   479→    # Auth
   480→    jwt_secret: SecretStr
   481→    jwt_algorithm: str = "HS256"
   482→    access_token_ttl_minutes: int = 15
   483→    refresh_token_ttl_days: int = 7
   484→
   485→    # Rate limiting
   486→    rate_limit_default: str = "60/minute"
   487→    rate_limit_chatbot: str = "10/minute"
   488→
   489→    # CORS (explicit allowlist, never "*")
   490→    cors_allowed_origins: list[str] = []
   491→
   492→    # ML serving
   493→    ml-serving_url: str | None = None
   494→    eta_model_version: str | None = None

   498→
   499→Per-country overrides (fare tables, capacity defaults, language) live in a `country_config`
   500→table, not in code. All magic numbers from the original (radii, speeds, thresholds, timeouts)
   501→move here.
   502→
   503→### 5.5 Background workers
   504→
   505→Time-series tasks (telemetry aggregation, retention pruning, demand retraining triggers,
   506→alert escalation timers) run as **Celery** workers (or a lightweight async task queue like
   507→ARQ) backed by Redis — not as daemon threads inside the FastAPI process (the original's
   508→`demo_simulator` daemon thread). This separates the request path from background work and
   509→allows independent scaling.

   514→
   515→### 6.1 PostgreSQL schema (core tables)
   516→
   517→The original's 8 tables × 5 countries = 40 SQLite files collapse into a single PostgreSQL
   518→database with proper relational design:
   519→
   520→```
   521→countries (country_code PK, name, currency, default_language, fare_config JSONB, ...)
   522→operators (operator_id PK, country_code FK, name, license_no, status, created_at)
   523→vehicles  (vehicle_id PK, operator_id FK, plate_no, route_id FK, capacity, device_id FK, status)
   524→routes    (route_id PK, country_code FK, code, name, geometry GEOMETRY(LineString),
   525→           distance_km NUMERIC, fare_min NUMERIC, fare_per_km NUMERIC, region, status)
   526→route_points (point_id PK, route_id FK, seq INT, lat, lon, is_stop BOOLEAN, stop_name)
   527→devices   (device_id PK, cert_serial, hardware_model, firmware_version, model_version,
   528→           vehicle_id FK NULL, status, provisioned_at, last_heartbeat_at)
   529→
   530→telemetry_logs   (id PK BIGSERIAL, vehicle_id FK, device_id FK, timestamp TIMESTAMPTZ,
   531→                  lat, lon, accuracy_m, speed_kph, heading, occupancy INT, tier,
   532→                  boarded INT, alighted INT, signal_quality, schema_version, seq INT)
   533→                  PARTITION BY RANGE (timestamp)  -- monthly partitions for scale
   534→
   535→vehicle_states   (vehicle_id PK, last_telemetry JSONB, last_seen_at, current_tier,
   536→                  current_route FK, online BOOLEAN)  -- latest state, hot-read
   537→

   539→                  acknowledged_at, acknowledged_by FK, resolved_at, resolution,

   541→

   543→

   545→                  intent, entities JSONB, language, at, pii_redacted BOOLEAN)

   554→
   555→### 6.2 Key design decisions
   556→
   557→- **Row-Level Security:** `country_code` on tenant-scoped tables; RLS policies enforce that a
   558→  PH-scoped token cannot read TH data. This replaces the original's file-per-country
   559→  isolation with something that actually scales.
   560→- **Partitioning:** `telemetry_logs` is range-partitioned by month. Old partitions can be
   561→  detached, archived, or dropped without affecting live queries. This is the only way to
   562→  handle the write volume (every vehicle, every 3 seconds, indefinitely).
   563→- **PostGIS:** `routes.geometry` is a `GEOMETRY(LineString)` with a spatial index. Route
   564→  deviation detection becomes a `ST_DWithin` query instead of the original's manual
   565→  haversine loop. Geofencing is a database primitive, not application code.
   566→- **No denormalized polylines:** the original stored `polyline_json` *and* `route_points`
   567→  rows (drift risk). Re-LoadSense stores geometry once (in `routes.geometry`) and derives
   568→  point lists on demand.
   569→- **Migrations:** Alembic migrations are versioned, peer-reviewed, and reversible. No
   570→  `ALTER TABLE IF COLUMN MISSING` hacks. CI applies migrations to a throwaway DB on every PR.
   571→- **Indexes:** every foreign key, every query filter column, every range-scan column. The
   572→  original had indexes on "the obvious columns" — Re-LoadSense has indexes on *the columns
   573→  the actual queries use*, validated by `EXPLAIN` in CI.
   574→
   575→### 6.3 Redis usage
   576→
   577→| Key pattern | Purpose | TTL |
   578→|---|---|---|
   579→| `vehicle:{id}:state` | Latest vehicle state (hot read) | 60s (refreshed on telemetry) |
   580→| `fleet:{country}:live` | Set of online vehicle IDs per country | 120s |
   581→| `route:{id}:polyline` | Cached polyline (avoid PostGIS round-trip) | 1h |
   582→| `eta:{vehicle}:{stop}` | Cached ETA prediction | 30s |
   583→| `rate:{ip}:{route}` | Rate limit token bucket | sliding window |
   584→| `session:{token}` | WebSocket session → subscriber mapping | session lifetime |
   585→| `pubsub:fleet:{country}` | Fleet update pub/sub channel | — |

   589→## 7. API design
   590→
   591→### 7.1 Versioning and conventions
   592→
   593→- All REST endpoints under `/api/v1/`. Breaking changes require `/v2/`.
   594→- Every endpoint declares `response_model`, `tags`, `summary`, `description`, `responses=`.
   595→- HTTP semantics: `201 Created` + `Location` for creation, `204 No Content` for deletion,
   596→  `200 OK` for reads and updates.
   597→- Pagination via `cursor` + `limit` (not offset) on list endpoints.
   598→- Errors are a consistent JSON shape: `{"error": {"code": "...", "message": "...",
   599→  "details": {...}, "request_id": "..."}}`. Internal exception text is **never** returned
   600→  (fixing the original's `str(exc)` leakage).
   601→- The OpenAPI spec is exported, committed, and diffed in CI. Breaking schema changes fail
   602→  the build.

   641→### 7.3 Input validation
   642→
   643→Every Pydantic schema uses constrained types:
   644→
   645→```python
   646→class TelemetryIngest(BaseModel):
   647→    schema_version: str = Field(pattern=r"^\d+\.\d+\.\d+$")
   648→    device_id: UUID4
   649→    vehicle_id: str = Field(min_length=1, max_length=32, pattern=r"^[A-Z0-9-]+$")
   650→    firmware_version: str = Field(pattern=r"^\d+\.\d+\.\d+$")
   651→    model_version: str = Field(pattern=r"^\d+\.\d+\.\d+$")
   652→    timestamp: datetime  # ISO 8601, not bare str
   653→    gps: GpsPayload
   654→    counts: CountsPayload
   655→    tier: Literal["available", "filling", "at_capacity", "overloaded"]
   656→    signal_quality: Literal["excellent", "good", "fair", "poor", "lost"]
   657→    seq: int = Field(ge=0)
   658→
   659→class GpsPayload(BaseModel):
   660→    lat: float = Field(ge=-90, le=90)
   661→    lon: float = Field(ge=-180, le=180)
   662→    accuracy_m: float = Field(ge=0, le=1000)
   663→    speed_kph: float = Field(ge=0, le=200)
   664→    heading: float = Field(ge=0, lt=360)
   665→
   666→class CountsPayload(BaseModel):
   667→    boarded: int = Field(ge=0, le=50)
   668→    alighted: int = Field(ge=0, le=50)
   669→    current_occupancy: int = Field(ge=0, le=100)
   670→```
   671→
   672→File uploads (`/routes/import`) are streaming-parsed with a hard size cap (e.g., 5 MB) and a
   673→row cap (e.g., 10,000 rows). The original read the entire upload into memory with no limit.

   677→## 8. Machine learning pipeline
   678→
   679→The original's ML layer was broken end-to-end (fake artifacts, target leakage, train/serve
   680→mismatch, `pickle.load` RCE, non-deterministic forecasts, no MLOps). Re-LoadSense builds a
   681→real MLOps stack.
   682→
   683→### 8.1 MLOps architecture
   684→
   685→```
   686→┌──────────────────────────────────────────────────────────────────────┐
   687→│                        ML PLATFORM                                   │
   688→│                                                                      │
   689→│  ┌──────────────┐   ┌──────────────┐   ┌──────────────────────────┐  │
   690→│  │ Feature      │   │ Training     │   │ Model Registry           │  │
   691→│  │ Store (Feast)│   │ pipeline     │   │ (MLflow)                 │  │
   692→│  │              │   │ (GitHub      │   │                          │  │
   693→│  │ feature defs │   │  Actions +   │   │  • versioned models      │  │
   694→│  │ train+serve  │   │  scheduled)  │   │  • artifacts (ONNX)      │  │
   695→│  │ parity       │   │              │   │  • metrics + params      │  │
   696→│  └──────┬───────┘   └──────┬───────┘   │  • promotion stages      │  │
   697→│         │                  │           │    (dev→staging→prod)    │  │
   698→│         ▼                  ▼           └────────────┬─────────────┘  │
   699→│  ┌──────────────────────────────────┐               │                │
   700→│  │  Data validation                 │               │                │
   701→│  │  (Great Expectations / Pandera)  │               │                │
   702→│  └──────────────────────────────────┘               │                │
   703→│                                                     ▼                │
   704→│  ┌──────────────────────────────────────────────────────────────┐   │
   705→│  │  Model Serving (Triton / BentoML / in-process ONNX Runtime)  │   │
   706→│  │  • loads ONNX from registry by version tag                   │   │
   707→│  │  • exposes gRPC/HTTP inference API                           │   │
   708→│  │  • Prometheus metrics (latency, throughput, errors)          │   │
   709→│  └──────────────────────────────────────────────────────────────┘   │
   710→│                                                                      │
   711→│  ┌──────────────────────────────────────────────────────────────┐   │
   712→│  │  Monitoring & Drift Detection (evidently / alibi-detect)     │   │
   713→│  │  • input drift (PSI, KS-test)                                │   │
   714→│  │  • prediction drift                                          │   │
   715→│  │  • performance drift (MAE over time)                         │   │
   716→│  │  • alerting on drift thresholds                              │   │
   717→│  └──────────────────────────────────────────────────────────────┘   │
   718→└──────────────────────────────────────────────────────────────────────┘
   719→```
   720→
   721→### 8.2 ETA model (fixed)
   722→
   723→**Problem with original:** target was a deterministic linear function of the features →
   724→meaningless evaluation. Only 4 of 74 routes in training data. `pickle.load` RCE.
   725→
   726→**Re-LoadSense approach:**
   727→
   728→- **Features (real):** `stop_index`, `time_of_day`, `day_of_week`, `is_holiday`,
   729→  `weather_condition`, `traffic_factor` (from real traffic API), `route_id` (embedded, not
   730→  one-hot, so unseen routes get a learned embedding neighborhood), `distance_to_stop_m`,
   731→  `current_speed_kph`, `historical_segment_travel_time_median`.
   732→- **Feature store:** Feast (or a typed inline feature module) defines features once; training
   733→  and inference both read from the same definitions → **no train-serve skew**.
   734→- **Target:** real observed travel times from `telemetry_logs` (not a synthetic formula).
   735→- **Evaluation:** time-based split (train on weeks 1–3, validate on week 4, test on week 5).
   736→  Held-out routes (train on routes A–M, test on N–Z) to measure generalization. Baseline
   737→  comparison: "predict the route's median ETA" — the model must beat this to ship.
   738→- **Serialization:** ONNX (not pickle). Loaded via `onnxruntime.InferenceSession` — no
   739→  arbitrary code execution.
   740→- **Versioning:** every model has an MLflow version tag. The inference client requests a
   741→  specific version (pinned in config) with fallback to a "champion" alias.
   742→- **Monitoring:** Prometheus metrics on `/api/v1/eta` (latency p50/p95/p99, error rate).
   743→  evidently checks for input drift (feature distribution shift) and prediction drift daily.
   744→  Drift beyond threshold triggers a retraining job and alerts the ML on-call.
   745→
   746→### 8.3 Demand forecast (fixed)
   747→
   748→**Problem with original:** trainer wrote a single file; backend read 5 per-country files (they
   749→never connected). Prophet on 5 days of data. Non-deterministic fallback (`random.uniform`).
   750→
   751→**Re-LoadSense approach:**
   752→
   753→- **Data:** real telemetry aggregated to hourly ridership per route, with a minimum of 8
   754→  weeks of data before the model trains (Prophet needs ≥2 weeks for daily seasonality; we
   755→  require 8 for robustness).
   756→- **Model:** Prophet (or NeuralProphet) per route, with `country_code`-aware holidays
   757→  (PH/ID/MY/TH/VN holiday calendars). Weekly + yearly seasonality enabled once enough data
   758→  exists.
   759→- **Output:** forecast stored in MLflow as a versioned artifact; the inference client loads
   760→  the champion version. The forecast is **cached in Redis** (TTL 1h) so `/api/v1/demand`
   761→  returns deterministic values within the cache window.
   762→- **Fallback:** if no model is loaded, the endpoint returns `{"source": "historical_mean",
   763→  "values": [...]}` — honest about what produced the numbers, and deterministic. Never
   764→  `random.uniform`.
   765→
   766→### 8.4 Chatbot (consolidated)
   767→
   768→**Problem with original:** 5 chatbot files, 2 dead, 1 with a live bug, multilingual claims
   769→unsupported.
   770→
   771→**Re-LoadSense approach:**
   772→
   773→- **Single `chatbot_service.py`** with a strategy interface: `HeuristicStrategy` (offline,
   774→  rule-based, always available) and `LlmStrategy` (external LLM via a provider SDK, optional).
   775→- **Strategy selection** is a config/feature-flag decision, not 5 copy-pasted files.
   776→- **The heuristic strategy is grounded:** it queries live fleet state and route metadata and
   777→  composes a response from real data — it never hallucinates route facts. (The original's
   778→  instinct to bypass the LLM for grounding was correct; Re-LoadSense formalizes it.)
   779→- **The LLM strategy** uses RAG: retrieve relevant fleet/route context, pass it to the LLM
   780→  with a strict system prompt, and post-process the response to ensure cited entities (route
   781→  codes, vehicle IDs) actually exist in the retrieved context. No ungrounded claims.
   782→- **i18n:** language detection on the query; the system prompt and response are localized.
   783→  The original's multilingual claim becomes real.
   784→- **PII redaction:** before persisting to `chatbot_queries`, the service redacts phone
   785→  numbers, email addresses, and other PII patterns. The original persisted raw queries.
   786→- **Observability:** every chatbot call emits a trace span (input, strategy used, output,
   787→  latency, tokens if LLM). Conversations are rate-limited per user.
   788→
   789→### 8.5 Reproducibility
   790→
   791→- All training scripts use **seeded** RNG (`random.seed`, `np.random.seed`,
   792→  `torch.manual_seed` if applicable). The original was unseeded everywhere.
   793→- Training data snapshots are versioned (DVC or a data registry) so a model version is
   794→  reproducible from its code commit + data version + seed.
   795→- `requirements.txt` / `pyproject.toml` pin exact versions with hashes. No `>=`.

   809→- **Client state:** Zustand (lightweight, no boilerplate).
   810→- **Forms:** React Hook Form + Zod (schema validation shared with backend where possible).
   811→- **Map:** `react-leaflet` (typed) or MapLibre GL JS for vector tiles + native dark mode.
   812→- **i18n:** i18next + react-i18next. Locales: en, fil, ceb, id, ms, th, vi.

   856→## 10. Security model
   857→
   858→Security lands in the first vertical slice — not "phase 2."
   859→
   860→### 10.1 Authentication
   861→
   862→- **Commuters/operators/admins:** email + password (bcrypt) or phone OTP, issued JWT access
   863→  tokens (15 min) + refresh tokens (7 days, rotated on use). Tokens are `HttpOnly`,
   864→  `SameSite=Strict`, `Secure` cookies for browser clients; bearer tokens for API clients.
   865→- **Edge devices:** mTLS with per-device client certificates issued during provisioning. Cert
   866→  serials are tracked in the `devices` table; revoked certs are blocked at the TLS handshake.
   867→- **API keys:** operators can issue scoped API keys for integrations (e.g., a fleet
   868→  management system). Keys are hashed at rest; scopes are enforced.
   869→
   870→### 10.2 Authorization (RBAC)
   871→
   872→| Role | Scope | Example capabilities |
   873→|---|---|---|
   874→| `commuter` | self | Read live fleet, routes, ETA; use chatbot; manage own profile |
   875→| `operator` | own operator's vehicles | Everything commuter can + read own fleet/alerts/demand, verify alerts, manage own vehicles/routes |
   876→| `admin` | own country | Everything operator can + manage users, devices, audit log within country |
   877→| `regulator` | own country (read-only, aggregated) | Read compliance dashboard, export reports |
   878→| `device` | own vehicle | POST telemetry, heartbeat |
   879→| `system` | cross-cutting | Internal service-to-service calls (e.g., retraining trigger) |
   880→
   881→Row-Level Security in PostgreSQL enforces scoping at the data layer, so an application bug
   882→cannot leak cross-tenant data.
   883→
   884→### 10.3 Input validation & output encoding
   885→
   886→- Every input validated via Pydantic constraints (§7.3).
   887→- Every output that includes user-supplied content is auto-escaped (React JSX by default).
   888→  `dangerouslySetInnerHTML` is banned except with DOMPurify sanitization.
   889→- SQL via parameterized ORM queries only (no raw string interpolation). The original's
   890→  `f"SELECT ... FROM {table}"` pattern in `database_status` is banned.
   891→
   892→### 10.4 Rate limiting
   893→
   894→- Token-bucket per IP + per API key, Redis-backed, configurable per route.
   895→- Stricter limits on expensive endpoints: `/chatbot` (10/min), `/places` (30/min),
   896→  `/trip-suggestions` (20/min).
   897→- Edge telemetry is rate-limited per device to prevent a malfunctioning device from flooding
   898→  the ingest pipeline.
   899→
   900→### 10.5 Security headers & transport
   901→
   902→- HTTPS everywhere (HSTS, `max-age=63072000; includeSubDomains; preload`).
   903→- CSP (`default-src 'self'`), `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`,
   904→  `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` restricted.
   905→- CORS: explicit allowlist per environment (`LOADSENSE_CORS_ALLOWED_ORIGINS`). Never `*`.
   906→- Cookies: `HttpOnly`, `Secure`, `SameSite=Strict`.
   907→
   908→### 10.6 Dependency & supply-chain security
   909→
   910→- All dependencies pinned with hashes (`pip install --require-hashes`,
   911→  `npm ci` with `package-lock.json`).
   912→- `pip-audit`, `safety`, `npm audit`, and `trivy` image scans in CI. New CVEs block the PR.
   913→- SBOM (CycloneDX) generated and published with each release.
   914→- Renovate/Dependabot for update PRs; security updates merged within 72h.
   915→
   916→### 10.7 Secrets management
   917→
   918→- No secrets in env vars in production. Secrets live in a vault (HashiCorp Vault or cloud
   919→  secret manager) and are injected at runtime.
   920→- `.env` files are dev-only and `.gitignore`d. `.env.example` documents required vars
   921→  without real values.
   922→- `jwt_secret`, `database_url`, `redis_url`, LLM API keys, mTLS CA keys — all vault-managed.
   923→
   924→### 10.8 Threat model (STRIDE)
   925→
   926→A STRIDE threat model is written and reviewed before each major release. It covers: spoofing
   927→(forged device certs, forged JWTs), tampering (telemetry injection, route editing), repudiation
   928→(audit log gaps), information disclosure (`str(exc)` leakage, cross-tenant reads, PII in
   929→chatbot logs), denial of service (rate limits, upload caps, partition pruning), elevation of
   930→privilege (RBAC bypass, RLS bypass). Each threat has a mitigation and an owner.
   931→

   934→## 11. Privacy & compliance
   935→
   936→The original had no Privacy Impact Assessment despite counting passengers from cameras. Re-
   937→LoadSense treats privacy as a launch blocker.
   938→
   939→### 11.1 Data minimization at the edge
   940→
   941→- **No images leave the device.** Camera frames are processed on-device and immediately
   942→  discarded. Only aggregate counts (`boarded`, `alighted`, `current_occupancy`, `tier`) are
   943→  telemetered.
   944→- **No bounding boxes or coordinates leave the device.** The original wrote per-frame
   945→  `centroid_x`/`centroid_y`/`density_zone` to a CSV — Re-LoadSense does not.
   946→- **No biometric data.** Occupancy counting does not use face recognition, gait analysis, or
   947→  any biometric. ByteTrack track IDs are ephemeral (reset on each trip / power cycle) and
   948→  never telemetered.
   949→
   950→### 11.2 Data retention
   951→
   952→- Raw `telemetry_logs`: retained 90 days hot, 1 year cold (partitioned, archivable).
   953→- `chatbot_queries`: PII-redacted, retained 30 days, then purged.
   954→- `operator_alerts` + `operator_feedback`: retained 2 years (regulatory compliance).
   955→- `audit_log`: retained 3 years.
   956→- Retention is enforced by scheduled partition-drop jobs (not manual `purge` calls that
   957→  nobody runs).
   958→
   959→### 11.3 Data subject rights
   960→
   961→- Commuters can request their `chatbot_queries` history (by session ID or authenticated user
   962→  ID) and request deletion.
   963→- The system supports right-to-access and right-to-erasure workflows per the Philippines Data
   964→  Privacy Act 2012 (and GDPR for any future EU expansion).
   965→
   966→### 11.4 Compliance matrix
   967→
   968→| Framework | Status |
   969→|---|---|
   970→| Philippines Data Privacy Act 2012 | Compliance matrix maintained; PIA completed before pilot |
   971→| ISO 27001 (information security) | Controls mapped; target certification post-pilot |
   972→| UN SDG 9 & 11 | Aligned (mission-level) |
   973→| ODbL (OpenStreetMap) | Attribution maintained; usage policy respected |
   974→| Nominatim/Photon usage policy | Valid contact User-Agent; rate limits respected |
   975→| LLM provider terms | Compliant usage; no PII sent to LLM (redacted pre-call) |
   976→
   977→### 11.5 Privacy Impact Assessment (PIA)
   978→
   979→A PIA is written before any pilot deployment, covering: data flows, lawful basis, data
   980→minimization, retention, subject rights, third-party sharing (none beyond LLM with
   981→redaction), risk assessment, and mitigations. The PIA is reviewed by a data protection
   982→officer (or equivalent) and published.

   986→## 12. Observability & operability
   987→
   988→### 12.1 Logging
   989→
   990→- **structlog** with JSON output, correlation IDs (`request_id` propagated via headers), and
   991→  structured fields (actor, action, resource, latency, status).
   992→- Log levels: `DEBUG` (dev only), `INFO` (default), `WARNING`, `ERROR`, `CRITICAL`.
   993→- No `print()` in production code (the original used `print()` throughout). Lint enforces
   994→  this.
   995→- Logs shipped to Loki (or equivalent) with retention policies.
   996→
   997→### 12.2 Metrics
   998→
   999→- **Prometheus** `/metrics` endpoint with RED metrics per route (Rate, Errors, Duration) plus
  1000→  business metrics: telemetry ingest rate, alert raise/resolve rates, ETA prediction latency,
  1001→  model inference latency, WebSocket subscriber count, device heartbeat lag.
  1002→- **Grafana** dashboards per service: API, edge ingest, ML serving, WebSocket gateway,
  1003→  database, Redis.
  1004→- Alerting rules: error rate > 1% for 5 min, p95 latency > 500ms for 5 min, ingest rate
  1005→  drop > 50%, model drift exceeded.
  1006→
  1007→### 12.3 Tracing
  1008→
  1009→- **OpenTelemetry** auto-instrumentation for FastAPI, asyncpg, Redis, HTTP clients.
  1010→- Distributed traces from edge → ingest → service → DB → ML serving, correlated by
  1011→  `trace_id`.
  1012→- Sampling: 100% of errors, 10% of successes (configurable).
  1013→
  1014→### 12.4 Error tracking
  1015→
  1016→- **Sentry** for frontend and backend exception capture with source maps.
  1017→- Release tracking: each deploy is tagged in Sentry so regressions are attributable.
  1018→
  1019→### 12.5 Health checks
  1020→
  1021→- `GET /health` → liveness (process alive, event loop responsive). Returns 200 fast.
  1022→- `GET /ready` → readiness (DB reachable, Redis reachable, model loaded). Returns 503 if
  1023→  any dependency is down; load balancer pulls the instance out of rotation.
  1024→- Docker Compose and k8s use these for healthcheck/readiness probes.
  1025→
  1026→### 12.6 Runbooks
  1027→
  1028→A `runbooks/` directory contains a markdown runbook per operational scenario: deploy, rollback,
  1029→rotate JWT secret, rotate mTLS CA, retrain ETA model, handle device cert revocation, respond
  1030→to a data subject access request, handle a drift alert, handle an ingest outage. Each runbook
  1031→has an owner and is tested quarterly.
  1032→
  1033→### 12.7 SLOs & error budgets
  1034→
  1035→| Service | SLI | SLO target | Error budget |
  1036→|---|---|---|---|
  1037→| API (read) | availability + p95 < 300ms | 99.9% | 43m/month |
  1038→| API (write) | availability + p95 < 500ms | 99.5% | 3.6h/month |
  1039→| Edge ingest | availability + drop rate < 0.01% | 99.9% | 43m/month |
  1040→| ETA prediction | p95 < 100ms | 99% | 7.2h/month |
  1041→| WebSocket | connection success rate | 99.5% | 3.6h/month |
  1042→
  1043→Error budget exhaustion freezes feature work on that service until the budget is recovered.

  1047→## 13. Scalability & deployment topology
  1048→
  1049→### 13.1 Horizontal scaling
  1050→
  1051→- FastAPI behind a load balancer; scale horizontally by adding replicas. Live state is in
  1052→  Redis (shared), so replicas are stateless.
  1053→- WebSocket gateway scales horizontally with Redis pub/sub adapter.
  1054→- Edge ingest scales independently (it's the highest-write-rate service).
  1055→- ML serving scales independently (GPU vs CPU pools).
  1056→- Background workers (Celery) scale by queue depth.
  1057→
  1058→### 13.2 Database scaling
  1059→
  1060→- PostgreSQL with read replicas for read-heavy paths (fleet, routes, demand).
  1061→- `telemetry_logs` partitioned by month; old partitions archived to object storage (S3-
  1062→  compatible) and queryable via a foreign data wrapper if needed.
  1063→- Connection pooling via PgBouncer.
  1064→- Redis cluster for cache + pub/sub at scale.
  1065→
  1066→### 13.3 Deployment
  1067→
  1068→- **Dev:** Docker Compose (app + Postgres + Redis + MLflow + Prometheus + Loki + Grafana).
  1069→- **Staging:** k8s (kind or cloud-managed), mirroring prod at smaller scale.
  1070→- **Prod:** k8s (cloud-managed EKS/GKE/AKS or self-hosted), Helm chart, GitOps (Argo CD or
  1071→  Flux) for declarative deploys, canary releases via Argo Rollouts.
  1072→
  1073→### 13.4 CI/CD
  1074→
  1075→GitHub Actions pipelines:
  1076→
  1077→1. **On PR:** lint (ruff + eslint), type-check (mypy + tsc), unit tests (pytest + vitest),
  1078→   integration tests (test DB on tmpfs), coverage gate (≥80% backend, ≥70% frontend), SAST
  1079→   (bandit + semgrep), dependency scan (pip-audit + npm audit), docs-drift check, OpenAPI
  1080→   schema diff.
  1081→2. **On merge to main:** build Docker images (multi-stage, non-root), push to registry with
  1082→   SBOM, deploy to staging, run e2e (Playwright) + load tests (k6) + contract tests
  1083→   (schemathesis).
  1084→3. **On tag:** promote image to prod via Argo Rollouts canary (10% → 50% → 100% with auto
  1085→   rollback on SLO breach).
  1086→4. **Scheduled (daily):** retrain ETA model if drift detected; regenerate demand forecast;
  1087→   partition pruning; dependency scan; SBOM publish.
  1088→
  1089→### 13.5 Backup & disaster recovery
  1090→
  1091→- PostgreSQL: automated daily snapshots + point-in-time recovery (WAL archiving). Restore
  1092→  tested quarterly.
  1093→- Redis: AOF persistence; treated as ephemeral (rebuildable from PostgreSQL).
  1094→- MLflow artifacts: backed up to object storage with versioning.
  1095→- RPO: 15 minutes (WAL). RTO: 1 hour. Tested via quarterly game-day exercises.

  1137→## 15. Open questions & explicit non-goals
  1138→
  1139→### 15.1 Open questions (to resolve during implementation)
  1140→
  1141→1. **LLM provider for the chatbot:** local Ollama (privacy-preserving, no data leaves the
  1142→   cloud VPC) vs. a hosted provider (lower latency, higher quality). Decision deferred to
  1143→   pilot based on latency/quality benchmarks and privacy review.
  1144→2. **Edge hardware unit economics:** Raspberry Pi 5 + camera + GPS + modem + enclosure cost
  1145→   vs. operator willingness to pay. Needs a pilot business model.
  1146→3. **Map tile provider:** self-hosted OSM tiles (cost, control) vs. a commercial provider
  1147→   (Mapbox/MapTiler — cost, ToS). Decision deferred to pilot scale.
  1148→4. **Regulator data sharing agreement:** what exactly can LGUs see, under what agreement, and
  1149→   with what retention? Needs legal review before the regulator dashboard ships.
  1150→5. **Payment integration:** if operators pay a subscription, what payment provider (Stripe
  1151→   isn't ideal for PH; GCash/Maya are local)? Deferred to post-pilot monetization.
  1152→
  1153→### 15.2 Explicit non-goals (for v1 / pilot)
  1154→
  1155→- **Face recognition or any biometric identification.** Occupancy counting counts heads, not
  1156→  identities. This is a hard ethical line.
  1157→- **Public-facing real-time vehicle tracking API without operator consent.** Live vehicle
  1158→  positions are shared only with the operator and (anonymized/aggregated) with commuters via
  1159→  the app. A public firehose is not a goal.
  1160→- **Driver discipline automation.** Alerts are advisory and operator-first; the system never
  1161→  automatically fines or penalizes drivers. Due-process review is required before any
  1162→  disciplinary use — same principle as the original, formalized.
  1163→- **Replacing jeepneys.** Re-LoadSense retrofits; it does not require fleet replacement.
  1164→- **Coverage outside ASEAN in v1.** The concept is ASEAN-first; EU/Africa expansion is a
  1165→  future consideration with separate compliance work.
  1166→- **Real-time traffic API in v1.** OpenWeatherMap and a traffic API are wired as features
  1167→  arrive, but the v1 pilot can run without them (ETA uses historical medians + GPS speed).
